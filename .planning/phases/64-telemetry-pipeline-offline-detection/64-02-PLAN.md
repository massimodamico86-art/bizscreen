
**Part 2: Create evaluate_and_alert_offline_devices function**

Create a `SECURITY DEFINER` function (no auth.uid() references — runs from cron without auth context):

```sql
CREATE OR REPLACE FUNCTION evaluate_and_alert_offline_devices(
  p_threshold_minutes INTEGER DEFAULT 5
)
RETURNS JSONB
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
```

The function does two things:

**2a. Detect and alert offline devices:**
- Query `tv_devices` for devices where `is_online = true` AND `last_seen < NOW() - (p_threshold_minutes || ' minutes')::INTERVAL`
- For each: mark `is_online = false` on the device row
- Insert alert into `alerts` table with:
  - `type = 'device_offline'`
  - `severity` escalation: info (<15 min), warning (15-59 min), critical (>=60 min)
  - `title`: 'Device "DeviceName" is offline'
  - `message`: 'No heartbeat for X minutes'
  - `device_id`: the device UUID
  - `tenant_id`: from the tv_devices row
  - `meta` JSONB: `{ device_name, minutes_offline, last_heartbeat, detected_at }`
- Use `ON CONFLICT ON CONSTRAINT idx_alerts_coalesce DO UPDATE` to coalesce duplicates:
  - Increment `occurrences`
  - Update `last_occurred_at`
  - Escalate severity (use GREATEST or CASE to keep the higher severity)
  - Merge meta

**2b. Auto-resolve alerts for recovered devices:**
- UPDATE `alerts` SET `status = 'resolved'`, `resolved_at = NOW()`, `resolution_notes = 'Auto-resolved: device resumed heartbeats'`
- WHERE `type = 'device_offline'` AND `status IN ('open', 'acknowledged')` AND `device_id IN (SELECT id FROM tv_devices WHERE is_online = true AND last_seen >= NOW() - (p_threshold_minutes || ' minutes')::INTERVAL)`

Return a JSON summary: `{ raised: N, resolved: N, evaluated_at: timestamp }`

**Part 3: Extend update_device_status for instant auto-resolve**

Read the current `update_device_status` function from migration 149 (the Plan 01 version). Use `CREATE OR REPLACE FUNCTION` to add auto-resolve logic AFTER the existing UPDATE tv_devices statement:

```sql
-- Auto-resolve any open device_offline alerts for this device
UPDATE alerts
SET status = 'resolved',
    resolved_at = NOW(),
    resolution_notes = 'Auto-resolved: device resumed heartbeats'
WHERE device_id = p_device_id
  AND type = 'device_offline'
  AND status IN ('open', 'acknowledged');
```

This gives dual-path resolution: instant on heartbeat (here) + periodic sweep (cron). The heartbeat-side resolve is cheap — it's an indexed lookup by device_id + type + status.

IMPORTANT: The function override MUST preserve ALL existing parameters and behavior from migration 149 (including p_metrics, last_seen, is_online, player_version, cached_content_hash, needs_screenshot_update). Copy the full function body and ADD the auto-resolve block.

**Part 4: Schedule the cron job**
```sql
SELECT cron.schedule(
  'evaluate-offline-devices',
  '*/2 * * * *',
  $$ SELECT evaluate_and_alert_offline_devices(5); $$
);
```

Every 2 minutes, evaluate with a 5-minute threshold. Worst case detection time: ~7 minutes (device goes offline right after a check, next check at 2 min finds only 2 min stale, check after that at 4 min finds 4 min stale, check at 6 min finds 6 min > 5 min threshold = detected). Typical detection: 5-6 minutes.

- The function runs as superuser via pg_cron — do NOT reference `auth.uid()`.
- Use `p_threshold_minutes` parameter (default 5) matching the existing OFFLINE_THRESHOLD constant in playerService.js.
