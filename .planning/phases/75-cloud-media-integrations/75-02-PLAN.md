**OneDrive service** (`src/services/cloud/oneDriveService.js`):

Use `createScopedLogger('OneDriveService')`. Import from `./cloudOAuthService.js`.

Config constants:
- `VITE_MICROSOFT_CLIENT_ID` (env var — shared between OneDrive and SharePoint since both use Microsoft identity)
- OAuth URL: `https://login.microsoftonline.com/common/oauth2/v2.0/authorize`
- Token URL: `https://login.microsoftonline.com/common/oauth2/v2.0/token`
- Graph API URL: `https://graph.microsoft.com/v1.0`
- Scopes: `Files.Read.All offline_access`

Functions (all named exports):
- `startOneDriveOAuth()` — PKCE + state, redirect to Microsoft OAuth. Redirect URI: `${origin}/auth/cloud/callback?provider=onedrive`
- `handleOneDriveCallback(code, state)` — validate state, exchange code for tokens (POST to token URL with client_id, code, code_verifier, redirect_uri, grant_type, scope)
- `isOneDriveConnected()` — check token via cloudOAuthService
- `getOneDriveToken()` — get/refresh token. Refresh uses scope param too.
- `disconnectOneDrive()` — clear tokens
- `listOneDriveFiles({ folderId = 'root', pageToken = null, query = '' })` — GET `/me/drive/items/${folderId}/children?$select=id,name,size,lastModifiedDateTime,file,folder,@microsoft.graph.downloadUrl&$top=50`. If pageToken, use `@odata.nextLink`. If query, use `/me/drive/root/search(q='${query}')` instead. Returns `{ files: [...], nextPageToken }`. Map items to `{ id, name, isFolder, size, modified, thumbnailUrl: null, mimeType: item.file?.mimeType, downloadUrl: item['@microsoft.graph.downloadUrl'] }`.
- `getOneDriveDownloadUrl(itemId)` — GET `/me/drive/items/${itemId}` and return `@microsoft.graph.downloadUrl` from response.
- Internal `onedriveFetch(url, options)` — wrapper adding Authorization Bearer header

**SharePoint service** (`src/services/cloud/sharePointService.js`):

Use `createScopedLogger('SharePointService')`. Import from `./cloudOAuthService.js`.

Config constants:
- `VITE_MICROSOFT_CLIENT_ID` (same as OneDrive — both use Microsoft identity platform)
- Same OAuth/token URLs as OneDrive
- Graph API URL: `https://graph.microsoft.com/v1.0`
- Scopes: `Sites.Read.All Files.Read.All offline_access`

Functions (all named exports):
- `startSharePointOAuth()` — PKCE + state, redirect to Microsoft OAuth with SharePoint scopes. Redirect URI: `${origin}/auth/cloud/callback?provider=sharepoint`
- `handleSharePointCallback(code, state)` — validate state, exchange code for tokens
- `isSharePointConnected()` — check token via cloudOAuthService
- `getSharePointToken()` — get/refresh token
- `disconnectSharePoint()` — clear tokens
- `listSharePointSites({ query = '' })` — GET `/sites?search=${query || '*'}` returns list of sites the user has access to. Returns `{ sites: [{ id, name, displayName, webUrl }] }`.
- `listSharePointFiles({ siteId, folderId = 'root', pageToken = null, query = '' })` — GET `/sites/${siteId}/drive/items/${folderId}/children` with same select fields as OneDrive. If query, use `/sites/${siteId}/drive/root/search(q='${query}')`. Returns `{ files: [...], nextPageToken }`.
- `getSharePointDownloadUrl(siteId, itemId)` — GET `/sites/${siteId}/drive/items/${itemId}` and return download URL.
- Internal `sharepointFetch(url, options)` — wrapper adding Authorization Bearer header

Note: OneDrive and SharePoint share `VITE_MICROSOFT_CLIENT_ID` but use separate token storage keys ('onedrive' vs 'sharepoint') in cloudOAuthService so they maintain independent sessions.

**Google Photos service** (`src/services/cloud/googlePhotosService.js`):

Use `createScopedLogger('GooglePhotosService')`. Import from `./cloudOAuthService.js`.

Config constants:
- `VITE_GOOGLE_PHOTOS_CLIENT_ID` (env var — separate from Google Drive since different API/scopes)
- OAuth URL: `https://accounts.google.com/o/oauth2/v2/auth` (same as Drive)
- Token URL: `https://oauth2.googleapis.com/token` (same as Drive)
- API URL: `https://photoslibrary.googleapis.com/v1`
- Scopes: `https://www.googleapis.com/auth/photoslibrary.readonly`

Functions (all named exports):
- `startGooglePhotosOAuth()` — PKCE + state, redirect to Google OAuth with Photos scopes. Redirect URI: `${origin}/auth/cloud/callback?provider=gphotos`
- `handleGooglePhotosCallback(code, state)` — validate state, exchange code for tokens
- `isGooglePhotosConnected()` — check token via cloudOAuthService
- `getGooglePhotosToken()` — get/refresh token
- `disconnectGooglePhotos()` — clear tokens
- `listGooglePhotosAlbums({ pageToken = null })` — GET `/albums?pageSize=50` (with pageToken if provided). Returns `{ albums: [{ id, title, mediaItemsCount, coverPhotoBaseUrl }], nextPageToken }`.
- `listGooglePhotosMedia({ albumId = null, pageToken = null, pageSize = 50 })` — POST `/mediaItems:search` with body `{ albumId, pageSize, pageToken }` (if no albumId, lists all media). Returns `{ mediaItems: [{ id, filename, mimeType, baseUrl, mediaMetadata: { width, height, creationTime } }], nextPageToken }`. Note: baseUrl needs `=w{width}-h{height}` suffix for sized image or `=d` for download.
- `getGooglePhotosDownloadUrl(baseUrl)` — returns `${baseUrl}=d` (download variant of baseUrl). Unlike other providers, Google Photos doesn't use file IDs for download — the baseUrl from listing IS the access URL.
- Internal `gphotosFetch(url, options)` — wrapper adding Authorization Bearer header

Important: Google Photos API uses POST for mediaItems:search (not GET). The baseUrl from media item listings is temporary (valid ~60 min), so getGooglePhotosDownloadUrl should be called just before downloading. Include `access_type=offline` and `prompt=consent` in OAuth URL to ensure refresh token is returned.
