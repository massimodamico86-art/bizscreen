<requirement_coverage_notes>
**CLOCK-06 / WTHR-03 — Editor vs Player timezone design decision:**

The roadmap success criterion states: "Layout editor preview renders clock/weather widgets using the screen's assigned timezone (not just the element-saved timezone)."

In the **player** path, this is already correct — ViewPage passes `content.screen?.timezone` through LayoutRenderer -> ZonePlayer -> widgets. The actual screen timezone is always used at runtime.

In the **editor** path, a layout is screen-agnostic: it can be assigned to any screen, and there is no inherent "screen timezone" available. As documented in 63-RESEARCH.md (Open Question 1), the editor's preview timezone source is fundamentally different from the player's. The practical and architecturally correct default is **browser timezone** (`Intl.DateTimeFormat().resolvedOptions().timeZone`).

**What this plan actually closes:** The editor currently passes `element.timezone` (typically `undefined` or `'screen'`) with no screen timezone to resolve against — so `resolveTimezone('screen', undefined)` silently falls back to browser timezone anyway, but through an unintended code path. This plan explicitly threads browser timezone as the `timezone` prop, making the resolution intentional and correct. The gap being closed is: "timezone prop is threaded at all" (vs the current `undefined`), not "editor uses the actual screen timezone."

The player path remains unchanged and will always use the real screen's IANA timezone.
</requirement_coverage_notes>
