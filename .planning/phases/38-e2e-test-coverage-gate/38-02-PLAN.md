<task type="auto">
  <name>Task 1: Run baseline, triage failures, fix or skip</name>
  <files>tests/e2e/*.spec.js, .planning/phases/37-e2e-test-stabilization/SKIPPED-TESTS.md</files>
  <action>
  This task is iterative. Follow this process:

  **Step 1: Establish baseline.**
  Run the full test suite once with JSON reporter to establish current pass rate:
  ```bash
  PLAYWRIGHT_JSON_OUTPUT_FILE=test-results/e2e-results.json npx playwright test --reporter=json,list --workers=1
  ```
  Parse the JSON output to get: total passed, failed, skipped, pass rate. Record this as the baseline.

  **Step 2: Analyze failures.**
  From the JSON results, extract all failing tests. For each failure, categorize:
  - **Auth issue** -- test uses wrong auth pattern or credentials
  - **Selector mismatch** -- test looks for UI elements that don't match current app
  - **Timing issue** -- test has race conditions (should have been caught in Phase 37)
  - **App bug** -- test correctly identifies a real application bug
  - **Infrastructure** -- test fails due to external service, DB connection, etc.
  - **Test design** -- test assumptions don't match current app behavior

  **Step 3: Fix what's fixable (fix first, skip as last resort per user decision).**
  For each failing test, attempt to fix in this priority order:
  1. **Selector fixes** -- If a test looks for the wrong element name/text, update the selector to match current UI. These are quick wins. Common issues from Phase 37:
     - template-marketplace.spec.js: "Marketplace" vs "Templates" navigation
     - template-packs.spec.js: heading text mismatch, "Layouts" button not found
     - playlist-template.spec.js: selector timeouts
  2. **Auth pattern fixes** -- If a test uses manual login but should use storage state, convert it. Follow the patterns established in Phase 37 (storage state files: playwright/.auth/client.json, admin.json, superadmin.json).
  3. **Wait pattern fixes** -- If a test has timing issues, apply Phase 37 patterns: element.waitFor(), waitForLoadState('domcontentloaded'), Promise.race for loading states.
  4. **Project-specific skips** -- If a test only makes sense for one project (e.g., client-only test running on admin project), add `test.skip(testInfo.project.name !== 'chromium', 'Only runs on chromium project')`.

  **Step 4: Skip unfixable tests.**
  Only skip tests that fall into valid skip reasons (per user decision):
  - Real app bugs that would require app code changes (not test code changes)
  - Infrastructure dependencies (external services, env requirements)

  For each skip, add `test.skip()` or `test.describe.skip()` with a clear reason string.

  **Step 5: Re-run and iterate.**
  After fixes, run the gate script: `node scripts/e2e-gate.cjs`
  If pass rate is still below 90%, go back to Step 2 with the new failures.
  Continue until 90%+ is achieved.

  **Step 6: Update SKIPPED-TESTS.md.**
  Add a new "Phase 38 Updates" section to `.planning/phases/37-e2e-test-stabilization/SKIPPED-TESTS.md` documenting:
  - Any new tests skipped (with reason and date)
  - Any previously skipped tests that were re-evaluated and fixed
  - Any previously skipped tests that remain skipped (confirmed still valid)
  - Updated summary counts

  **Important constraints:**
  - Use `--workers=1` for consistent results (Phase 37 recommendation)
  - Run Chromium only (all 3 chromium projects: chromium, chromium-admin, chromium-superadmin)
  - Do NOT add new tests -- only fix or skip existing ones
  - Do NOT modify application source code -- only test files
  - Keep track of which tests were fixed vs skipped for the coverage report
  </action>
