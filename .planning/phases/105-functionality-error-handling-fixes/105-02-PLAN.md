```js
async function fetchPreviewContent(token) {
  const res = await fetch(`${API_BASE}/api/preview/${token}`);
  if (!res.ok) {
    const error = await res.json().catch(() => ({ error: 'Failed to load preview' }));
    throw new Error(error.error || 'Failed to load preview');
  }
  return res.json();
}
```

When an invalid token is used, the API endpoint may not exist (returns Vite's HTML 404 page).
The `res.ok` is false, and `res.json()` tries to parse HTML as JSON, which throws
"Unexpected token '<', '<!DOCTYPE '... is not valid JSON".
The .catch() on line 46 should handle this, but the SUCCESS path (line 49 `return res.json()`)
also has no protection -- if the endpoint returns an HTML page with status 200, it will crash.

Wait -- re-reading the code: line 46 DOES have `.catch(() => ({ error: 'Failed to load preview' }))`.
So the `!res.ok` path is actually handled. The problem must be the SUCCESS path at line 49:
if the API returns a 200 with HTML (e.g., the SPA index.html as a catch-all), then
`res.json()` will throw "Unexpected token '<'" with no catch.

The fix: wrap the success-path `res.json()` in a try/catch or add content-type checking.

**Fix fetchPreviewContent (line 43-50) to handle non-JSON responses on both error and success paths:**

The current code has a `.catch()` on the error path's `res.json()`, but the success path (line 49) calls `res.json()` with no protection. If the server returns HTML with a 200 status (e.g., SPA catch-all route returning index.html), the JSON parse will crash with "Unexpected token '<'".

Replace the entire `fetchPreviewContent` function:

```js
async function fetchPreviewContent(token) {
  const res = await fetch(`${API_BASE}/api/preview/${token}`);

  // Check Content-Type to avoid parsing HTML as JSON
  const contentType = res.headers.get('content-type') || '';

  if (!res.ok) {
    // Try to extract error message from JSON response, fall back to generic message
    if (contentType.includes('application/json')) {
      const error = await res.json().catch(() => ({}));
      throw new Error(error.error || 'This preview link is invalid or has expired.');
    }
    throw new Error('This preview link is invalid or has expired.');
  }

  // Verify the successful response is actually JSON (not an HTML catch-all)
  if (!contentType.includes('application/json')) {
    throw new Error('This preview link is invalid or has expired.');
  }

  // Parse the JSON response with safety catch
  try {
    return await res.json();
  } catch {
    throw new Error('This preview link is invalid or has expired.');
  }
}
```

This handles ALL edge cases:
1. **Server returns 404 with HTML** -- `!res.ok` is true, content-type is not JSON, throws clean error
2. **Server returns 404 with JSON error** -- `!res.ok` is true, parses JSON to extract error message
3. **Server returns 200 with HTML (SPA catch-all)** -- `res.ok` is true but content-type is HTML, throws clean error
4. **Server returns 200 with valid JSON** -- normal success path, works as before
5. **Server returns 200 with malformed JSON** -- caught by try/catch, throws clean error

The error message "This preview link is invalid or has expired." is user-friendly and matches the existing error UI which already shows "Preview Unavailable" heading and "This preview link may have expired or been revoked." description. The combined messages provide clear context.
