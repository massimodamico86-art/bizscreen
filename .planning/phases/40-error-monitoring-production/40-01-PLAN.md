<task type="auto">
  <name>Task 1: Install Sentry Vite plugin and configure source map uploads</name>
  <files>vite.config.js, src/utils/errorTracking.jsx</files>
  <action>
1. Install @sentry/vite-plugin as a dev dependency:
   `npm install @sentry/vite-plugin --save-dev`

2. Add import at top of vite.config.js:
   `import { sentryVitePlugin } from '@sentry/vite-plugin';`

3. Add `sentryVitePlugin()` as the LAST plugin in the plugins array (after visualizer). The plugin must be last so all other transforms complete first:
   ```
   sentryVitePlugin({
     org: process.env.SENTRY_ORG,
     project: process.env.SENTRY_PROJECT,
     authToken: process.env.SENTRY_AUTH_TOKEN,
     sourcemaps: {
       filesToDeleteAfterUpload: ['./dist/**/*.map'],
     },
   }),
   ```

4. Add `sourcemap: "hidden"` to the `build` config object. Use `"hidden"` (NOT `true`) -- "hidden" generates .map files but does NOT add `//# sourceMappingURL` comments to the bundle, preventing public exposure of source maps.

5. Remove the `release` property from `Sentry.init()` in `src/utils/errorTracking.jsx` (line 74: `release: import.meta.env.VITE_APP_VERSION || '1.0.0'`). The Vite plugin auto-injects the release at build time via Debug IDs. Keeping both would cause a release mismatch that breaks source map association. Add a comment explaining why release is not set manually:
   ```
   // release is auto-injected by @sentry/vite-plugin at build time.
   // Do NOT set it manually -- a mismatch breaks source map resolution.
   ```

IMPORTANT: The sentryVitePlugin will gracefully skip upload if SENTRY_AUTH_TOKEN is not set (e.g., local dev). It only uploads during production builds where the env vars are configured.
  </action>

<task type="auto">
  <name>Task 2: Add Sentry environment variables to CI deploy workflow</name>
  <files>.github/workflows/deploy.yml</files>
  <action>
In the deploy workflow `.github/workflows/deploy.yml`, the "Build application" step (line 74-79) already sets `VITE_APP_VERSION` and `VITE_BUILD_TIME` as env vars. Add the three Sentry env vars to this same step so the Vite plugin can upload source maps during CI builds:

```yaml
      - name: Build application
        id: build
        run: |
          npm run build
          echo "build_id=$(git rev-parse --short HEAD)" >> $GITHUB_OUTPUT
        env:
          VITE_APP_VERSION: ${{ github.sha }}
          VITE_BUILD_TIME: ${{ github.event.head_commit.timestamp }}
          SENTRY_AUTH_TOKEN: ${{ secrets.SENTRY_AUTH_TOKEN }}
          SENTRY_ORG: ${{ secrets.SENTRY_ORG }}
          SENTRY_PROJECT: ${{ secrets.SENTRY_PROJECT }}
```

These are NOT prefixed with `VITE_` because they are server-side build-time variables consumed by the Vite plugin process, not client-side variables embedded in the bundle. The secrets must be configured in GitHub repository settings (or Vercel environment if using Vercel's build).

NOTE: The Sentry Vite plugin gracefully handles missing env vars (no upload, no build failure). This means existing CI runs will continue to work even before the secrets are added.
  </action>

- The three SENTRY_ vars appear under the `env:` block of the "Build application" step
