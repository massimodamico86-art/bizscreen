**Create src/utils/supabaseErrorInterceptor.js (new file):**

This utility intercepts Supabase client-side query and RPC failures and enriches them with Sentry breadcrumbs and structured context. Since there is no backend server, "API error monitoring" means capturing Supabase PostgREST errors on the frontend.

```javascript
/**
 * Supabase Error Interceptor
 *
 * Wraps the Supabase client to automatically:
 * 1. Add breadcrumbs for ALL Supabase operations (success and failure)
 * 2. Capture exceptions with structured context on failures
 *
 * Supabase JS client returns { data, error } -- it does NOT throw.
 * So we intercept the response after each query resolves.
 */

import { addBreadcrumb, captureException, setContext } from './errorTracking.jsx';
```

Create and export a function `instrumentSupabaseClient(client)` that uses a JavaScript Proxy to wrap the Supabase client's `.from()` method. The approach:

1. **Proxy `client.from(table)`** -- return a proxied query builder that intercepts terminal methods (`.select()`, `.insert()`, `.update()`, `.delete()`, `.upsert()`). Each terminal method on a Supabase query builder returns a Promise-like (thenable). Wrap the `.then()` on the returned object to inspect the `{ data, error }` result.

2. **On EVERY Supabase call (success or failure), add a breadcrumb:**
   ```javascript
   addBreadcrumb('supabase', `${operation} ${table}`, {
     table,
     operation, // 'select', 'insert', 'update', 'delete', 'upsert'
     status: error ? 'error' : 'ok',
     ...(error && { errorCode: error.code, errorMessage: error.message }),
   });
   ```

3. **On FAILURE (when `error` is not null), capture an exception:**
   ```javascript
   const supabaseError = new Error(`Supabase ${operation} on "${table}" failed: ${error.message}`);
   supabaseError.name = 'SupabaseApiError';
   captureException(supabaseError, {
     table,
     operation,
     postgrestCode: error.code,       // e.g., 'PGRST116', '42501'
     httpStatus: error.status || error.statusCode,
     hint: error.hint,
     details: error.details,
     message: error.message,
   });
   ```

4. **Also proxy `client.rpc(fnName, params)`** using the same pattern. For RPC calls, the "table" context is the function name and the "operation" is 'rpc':
   ```javascript
   addBreadcrumb('supabase', `rpc ${fnName}`, {
     function: fnName,
     operation: 'rpc',
     status: error ? 'error' : 'ok',
     ...(error && { errorCode: error.code, errorMessage: error.message }),
   });
   ```

5. **Do NOT proxy `client.auth.*` or `client.storage.*`** -- these have different response shapes and auth errors are handled separately in AuthContext. Only proxy `.from()` and `.rpc()`.

6. **Implementation strategy using Proxy:** The cleanest approach that avoids monkey-patching:
   ```javascript
   export function instrumentSupabaseClient(client) {
     return new Proxy(client, {
       get(target, prop, receiver) {
         if (prop === 'from') {
           return (table) => {
             const builder = target.from(table);
             return wrapQueryBuilder(builder, table);
           };
         }
         if (prop === 'rpc') {
           return (fnName, params, options) => {
             const result = target.rpc(fnName, params, options);
             return wrapRpcResult(result, fnName);
           };
         }
         return Reflect.get(target, prop, receiver);
       },
     });
   }
   ```

7. **`wrapQueryBuilder(builder, table)`** returns a Proxy over the builder that intercepts the terminal methods. When a terminal method is called (e.g., `.select(columns)`), it calls the real method, then wraps the resulting thenable to inspect `{ data, error }` before passing it through. The builder also supports chaining (`.eq()`, `.order()`, `.limit()`, etc.) -- those should pass through unmodified.

   Key implementation detail: Supabase query builders chain with methods like `.eq()`, `.order()`, `.limit()`, `.single()`, `.maybeSingle()` that return the same builder. The terminal call is implicit -- the builder itself is a thenable (has `.then()`). So the interception point is wrapping `.then()`:

   ```javascript
   function wrapQueryBuilder(builder, table) {
     let operation = 'select'; // default, updated by intercept

     return new Proxy(builder, {
       get(target, prop) {
         // Track which operation type this is
         if (['select', 'insert', 'update', 'delete', 'upsert'].includes(prop)) {
           operation = prop;
         }

         const value = Reflect.get(target, prop);
         if (typeof value !== 'function') return value;

         // Wrap the method call
         return (...args) => {
           const result = value.apply(target, args);

           // If result is thenable (terminal), wrap it
           if (result && typeof result.then === 'function' && prop === 'then') {
             return result; // Don't double-wrap .then()
           }

           // If result is a builder (chainable), wrap it too
           if (result && typeof result === 'object' && result !== target) {
             return wrapQueryBuilder(result, table);
           }
           return result;
         };
       },
     });
   }
   ```

   ACTUALLY -- the simpler and more reliable approach: instead of deep Proxy nesting, override `.then()` on the final builder. Supabase builders are PromiseLike, so any `await` calls `.then()`. Wrap `.then()` to add the breadcrumb/capture logic:

   ```javascript
   function wrapQueryBuilder(builder, table) {
     let operation = 'unknown';

     const handler = {
       get(target, prop) {
         if (['select', 'insert', 'update', 'delete', 'upsert'].includes(prop)) {
           operation = prop;
         }

         const original = Reflect.get(target, prop);
         if (typeof original !== 'function') return original;

         if (prop === 'then') {
           // Intercept the await/then to capture results
           return (onFulfilled, onRejected) => {
             return original.call(target, (result) => {
               handleSupabaseResult(result, table, operation);
               return onFulfilled ? onFulfilled(result) : result;
             }, onRejected);
           };
         }

         // For chainable methods, return wrapped result
         return (...args) => {
           const result = original.apply(target, args);
           if (result && typeof result === 'object' && typeof result.then === 'function') {
             return wrapQueryBuilder(result, table);
           }
           return result;
         };
       },
     };

     return new Proxy(builder, handler);
   }
   ```

8. **`handleSupabaseResult({ data, error }, table, operation)`** is the core function that adds the breadcrumb and optionally captures the exception (as described in points 2 and 3 above).

9. **`wrapRpcResult(result, fnName)`** wraps the RPC thenable similarly to intercept `{ data, error }` and call `handleSupabaseResult` with operation='rpc' and table=fnName.

10. **Export `captureSupabaseError(error, context)`** as a standalone function that services can call manually if they want to add extra context beyond what the interceptor captures automatically. This is for edge cases where a service wants to add business-specific context:
    ```javascript
    export function captureSupabaseError(error, context = {}) {
      if (!error) return;
      const enriched = new Error(error.message || 'Supabase error');
      enriched.name = 'SupabaseApiError';
      captureException(enriched, {
        postgrestCode: error.code,
        httpStatus: error.status || error.statusCode,
        hint: error.hint,
        details: error.details,
        ...context,
      });
    }
    ```
