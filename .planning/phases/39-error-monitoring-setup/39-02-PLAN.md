<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 1: Verify Sentry DSN is configured and errors appear in dashboard</name>
  <what-built>
    Plan 01 wired the Sentry SDK into the application lifecycle:
    - initObservability() is called before React renders
    - React 19 error hooks forward errors to Sentry
    - React Router v7 integration provides route-aware tracing
    - User identity propagates to Sentry on login/logout
    - All error tracking imports consolidated to single Sentry-backed provider

    For this to work end-to-end, VITE_SENTRY_DSN must be set in the environment.
  </what-built>
  <how-to-verify>
    **Step 1: Configure Sentry DSN**
    - If you don't have a Sentry project: Go to https://sentry.io -> Create Project -> Select "React" -> Copy the DSN
    - Add to `.env.local`: `VITE_SENTRY_DSN=https://your-dsn@o123.ingest.sentry.io/456`
    - Also set: `VITE_ERROR_TRACKING_PROVIDER=sentry`
    - Also set: `VITE_SENTRY_DEBUG=true` (enables Sentry in dev mode for testing)

    **Step 2: Start the dev server**
    - Run `npm run dev`
    - Open the app in browser

    **Step 3: Verify initialization**
    - Open browser DevTools console
    - Look for: `[errorTracking] Sentry initialized` log message
    - If you see `[ErrorTracking] Sentry DSN not configured` instead, the DSN env var is not set correctly

    **Step 4: Trigger a test error**
    - Open browser console and run: `throw new Error('Sentry test error from BizScreen');`
    - OR navigate to a non-existent deep route and check if Sentry captures the navigation

    **Step 5: Check Sentry dashboard**
    - Go to your Sentry project dashboard
    - Look for the test error within 60 seconds
    - Verify the error event includes:
      - Route/URL context (current page)
      - Browser/OS context (auto-captured)
      - Tags: app=bizscreen, platform=web

    **Step 6: Verify user context (if logged in)**
    - Log into the app
    - Trigger another test error from console: `throw new Error('Sentry test with user context');`
    - Check Sentry dashboard - the new error should include:
      - User ID
      - User email
      - User role (in additional context)

    **Step 7: Remove debug flag**
    - Remove `VITE_SENTRY_DEBUG=true` from `.env.local` (Sentry will only be active in production after this)
    - Set `VITE_SENTRY_DSN` in your production environment (Vercel dashboard -> Environment Variables)
  </how-to-verify>
  <resume-signal>Type "approved" if errors appear in Sentry with user/route context, or describe any issues</resume-signal>
</task>
