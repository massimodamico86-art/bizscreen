Add three new Playwright projects for viewport testing to `playwright.config.js`. These are ADDITIONAL projects alongside the existing four (setup, chromium, chromium-admin, chromium-superadmin).

Add these projects AFTER the existing `chromium-superadmin` project in the `projects` array:

```javascript
// Viewport presets for responsive testing (Phase 92+)
// Tests opt-in to these by using --project flag or test.describe config
{
  name: 'mobile',
  use: {
    ...devices['Desktop Chrome'],
    viewport: { width: 375, height: 667 },
    storageState: 'playwright/.auth/client.json',
  },
  dependencies: ['setup'],
  testMatch: /.*responsive.*\.spec\.js/,
},
{
  name: 'tablet',
  use: {
    ...devices['Desktop Chrome'],
    viewport: { width: 768, height: 1024 },
    storageState: 'playwright/.auth/client.json',
  },
  dependencies: ['setup'],
  testMatch: /.*responsive.*\.spec\.js/,
},
{
  name: 'desktop',
  use: {
    ...devices['Desktop Chrome'],
    viewport: { width: 1440, height: 900 },
    storageState: 'playwright/.auth/client.json',
  },
  dependencies: ['setup'],
  testMatch: /.*responsive.*\.spec\.js/,
},
```

Key design decisions:
- `testMatch: /.*responsive.*\.spec\.js/` -- These projects only run on spec files with "responsive" in the name (Phase 107 will create these). This prevents them from tripling the entire test suite run time.
- Uses `devices['Desktop Chrome']` as the base for all three (keeps Chromium engine consistent).
- `viewport` override on each project sets the viewport size.
- All use client auth by default (responsive tests are primarily UI layout tests). Phase 107 can override per-test.
- Existing projects (chromium, chromium-admin, chromium-superadmin) remain UNCHANGED.

Also, do NOT change the global `screenshot: 'only-on-failure'` setting. The screenshotStep() helper from Plan 92-01 handles explicit screenshot capture. We don't want Playwright auto-capturing screenshots at every step (would be noisy and unorganized).

<task type="auto">
  <name>Task 2: Update CI workflow and gitignore</name>
  <files>.github/workflows/ci.yml, .gitignore</files>
  <action>
1. **Update `.github/workflows/ci.yml`** -- Add a screenshot artifact upload step in the `e2e` job, AFTER the existing "Upload test results" step and BEFORE the "Upload E2E gate results" step:

```yaml
      - name: Upload E2E screenshots
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: e2e-screenshots
          path: screenshots/
          retention-days: 14
          if-no-files-found: ignore
```

Key decisions:
- `if: always()` -- Upload screenshots even if tests fail (failure screenshots are the most valuable).
- `retention-days: 14` -- Longer than the 7-day report retention because screenshots serve as documentation evidence.
- `if-no-files-found: ignore` -- Don't fail the workflow if no screenshots exist (tests might not have generated any yet during the transition period).
- Artifact name: `e2e-screenshots` (distinct from `test-results` and `playwright-report`).

2. **Update `.gitignore`** -- Add `screenshots/` to the gitignore file. Place it after the existing `playwright/.auth/` entry since both are test-generated artifacts:

```
# E2E test screenshots (generated during test runs)
screenshots/
```

This prevents the screenshots directory from being committed to the repository. Screenshots are ephemeral test artifacts, regenerated on each run, and uploaded as CI artifacts instead.
  </action>
