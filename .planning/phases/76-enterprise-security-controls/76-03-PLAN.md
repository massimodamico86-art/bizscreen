  <action>
    **Fix 1 — AuthContext.jsx line 102:**
    Change the profiles SELECT from:
    ```
    .select('id, email, full_name, role, has_completed_onboarding')
    ```
    to:
    ```
    .select('id, email, full_name, role, has_completed_onboarding, tenant_id')
    ```
    This is the single root cause for all 3 verification gaps. The `profiles` table has a `tenant_id` column (confirmed by usage in layoutTemplateService.js and notificationDispatcherService.js) but AuthContext never fetches it. Adding it to the SELECT makes `userProfile.tenant_id` available throughout the app.

    **Fix 2 — EnterpriseSecurityPage.jsx line 252:**
    Change:
    ```
    await requestDataDeletion(userProfile?.id, email, deletionReason || null);
    ```
    to:
    ```
    await requestDataDeletion(userProfile?.tenant_id, email, deletionReason || null);
    ```
    The `requestDataDeletion` function in complianceService.js expects a tenant ID as its first argument (passed as `p_tenant_id` in the RPC call). Currently, the user's profile ID is passed instead, which would target the wrong tenant or fail. With Fix 1 applied, `userProfile.tenant_id` will now be defined.

    - Policy saving (line 269 guard `if (!tenantId)` now passes because `userProfile?.tenant_id` is defined)
    - Tenant data deletion (correct tenant ID passed to RPC)
