### SQL-level RLS fallback path (NEW — closes SC-1.b without test creds)

The fallback test uses a Supabase JS client constructed with `SUPABASE_SERVICE_ROLE_KEY`. Service-role normally bypasses RLS via the `service_role` Postgres role. To trigger the RLS policy, the test issues a multi-statement RPC / `execute_sql`-style call (project pattern: a one-time Postgres function `exec_sql_as_authenticated_role` exposed only when SUPABASE_SERVICE_ROLE_KEY is the caller — OR, more portably, a single-shot SQL statement sent through the Mgmt API endpoint `/v1/projects/{id}/database/query` with `Authorization: Bearer ${SUPABASE_ACCESS_TOKEN}`). The simplest portable shape is:

```sql
BEGIN;
SET LOCAL ROLE authenticated;
INSERT INTO template_drafts (svg_content, prompt, source, status)
VALUES ('<svg/>', 'phase-176-rls-sql-fallback', 'admin_upload', 'pending');
ROLLBACK;
```

When no JWT is present and the role is plain `authenticated`, `auth.uid()` returns NULL and `is_admin()` therefore returns FALSE — so the policy `is_admin() OR is_super_admin()` rejects the INSERT with SQLSTATE `42501` (insufficient_privilege) regardless of the underlying transaction's superuser origin.

This Phase project EXPOSES the Mgmt API endpoint via `SUPABASE_ACCESS_TOKEN` (Phase 175 Plan 06 pattern). The fallback test:
1. Reads `SUPABASE_ACCESS_TOKEN` (skip-guarded)
2. POSTs the SQL above to `https://api.supabase.com/v1/projects/${PROJECT_ID}/database/query`
3. Asserts the response contains an error with code `42501` OR a message containing `row-level security` / `permission denied`

If `SUPABASE_ACCESS_TOKEN` is unavailable (rare on a dev box that ran Phase 175), the fallback skips — and the SUMMARY records SC-1.b as DEFERRED with the explicit note that the policy IS in the DB per Plan 02 Smoke 3 (positive proof of the policy's existence) and Phase 177 will close the live-behavior gap by being the first phase to actually write into template_drafts.
