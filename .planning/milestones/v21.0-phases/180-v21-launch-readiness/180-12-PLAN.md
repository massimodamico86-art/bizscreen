Close the **two remaining open_gaps** surfaced by Plan 180-11 Branch E (180-VERIFICATION.md lines 50-66) so that Phase 180 reaches `status: passed` + `score: 11/11` and v21.0 is launch-ready:

1. **SC-5_v21.1** (axe-core heading-order violation, moderate severity) — Plan 180-07's `<h3>` → `<h4>` demotion on TemplateCard.jsx:136 eliminated the original gridcell+h3→h4 batch BUT created a new "heading-order" violation when StarterPacksStrip collapses (no `<h2>` "Starter Packs" + no PackCard `<h3>` present), producing the chain page `<h1>` "Templates" → card `<h4>` (skips `<h2>`, `<h3>`). axe-core's `heading-order` rule walks the full document tree even though `AxeBuilder.include('[role="grid"]')` scopes the violation REPORTING — meaning the page `<h1>` ancestor IS considered. The fix is **Option (a)** from VERIFICATION.md line 55: inject a continuous `<h2>` + `<h3>` chain between the `<h1>` and the card `<h4>` so heading order is h1 → h2 → h3 → h4 regardless of StarterPacksStrip state. The cleanest implementation is **sr-only** headings (visually hidden, semantically present) placed inside the scroll container in TemplateGalleryPage.jsx, just above the render-branch tree — they exist on every render path (loading / error / zero-content / favorites-empty / no-results / content) and never disappear.

  **Why not Option (b)** (revert TemplateCard `<h4>` to `<h3>` with `role="heading" aria-level="4"`): axe-core still walks the DOM heading-level attribute (the literal `<h3>` tag), so reverting to `<h3>` re-introduces Plan 180-07's original violation when PackCard `<h3>` is present AND keeps a heading-order skip (h1 → h3 skips h2) when StarterPacksStrip is empty. role="heading" aria-level doesn't override the tag-name-driven heading-order analysis.

  **Why not Option (c)** (`<div role="heading" aria-level="4">`): same root cause as (b) — the heading-order rule analyzes both `<hN>` tags AND elements with `role="heading"`, and an aria-level=4 without preceding aria-level=2/3 still skips. Plus div-with-role-heading inside a `role="gridcell"` parent may itself trigger new ARIA violations (children of gridcell expected to be focusable interactive content, not heading-shaped). Option (a) is strictly safer.

### Axe spec scoping behavior (why sr-only headings inside the scroll container fix it)

```js
// tests/e2e/template-gallery-axe.spec.js:45-49
const results = await new AxeBuilder({ page })
  .include('[role="grid"]')
  .analyze();
expect(results.violations).toEqual([]);
```

axe-core's `heading-order` rule analyzes the FULL document tree for heading-level ordering (not just elements inside the `.include()` scope) — the `.include()` filter only restricts WHICH elements get reported as violation TARGETS. The `<h1>` ancestor outside `[role="grid"]` is considered when computing the chain. Therefore: adding `<h2>` and `<h3>` ANYWHERE in the document between the `<h1>` and the `<h4>` card titles satisfies the rule — including outside `[role="grid"]`. Placing them inside the scroll container in TemplateGalleryPage.jsx (which is the parent of `[role="grid"]`) is therefore sufficient AND keeps the change minimal.
