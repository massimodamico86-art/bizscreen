## Surface 1: Pending Tab — Filter Chip Strip

### Layout

Render a `<div>` with class `flex flex-col gap-3 mb-4` above the draft list.
Inside, render three `<FilterChips>` rows stacked vertically:
- Row 1: Vertical filter
- Row 2: Template-type filter
- Row 3: Status filter

Each row has a `<span className="text-xs font-semibold text-gray-500 mr-2 shrink-0">` label
preceding the chip group (e.g. "Vertical:", "Type:", "Status:").

### Idle-State Primary Visual Anchor

On page load with zero drafts selected, the filter chip strip is the primary
visual anchor — it guides the admin's first interaction toward narrowing the
draft pool before selecting. The strip is always rendered at the top of the
Pending tab, even when the draft list is empty.

### Chip Options

**Vertical filter** — single-select, default `all`:
```
{ id: 'all',          label: 'All verticals' }
{ id: 'restaurants',  label: 'Restaurants' }
{ id: 'retail',       label: 'Retail' }
{ id: 'healthcare',   label: 'Healthcare' }
{ id: 'cross',        label: 'Cross-vertical' }
```

**Template-type filter** — single-select, default `all`:
```
{ id: 'all',                   label: 'All types' }
{ id: 'menu',                  label: 'Menu' }
{ id: 'promo',                 label: 'Promo' }
{ id: 'announcement',          label: 'Announcement' }
{ id: 'reminder',              label: 'Reminder' }
{ id: 'wayfinding',            label: 'Wayfinding' }
{ id: 'health_tip',            label: 'Health Tip' }
{ id: 'queue_status',          label: 'Queue Status' }
{ id: 'drive_thru',            label: 'Drive-Thru' }
{ id: 'waiting_room_ambient',  label: 'Waiting Room' }
{ id: 'emergency_alert',       label: 'Emergency Alert' }
```
Use `maxVisible={6}` so the list does not overflow. The overflow `More` chip
triggers the existing `FilterChips` dropdown; no new dropdown code required.

**Status filter** — single-select, default `all`:
```
{ id: 'all',                label: 'All statuses' }
{ id: 'pending',            label: 'Pending' }
{ id: 'needs_human_review', label: 'Needs review' }
```

### Chip Visual States

All three chip groups use `variant="default"` (gray-900 active):

| State | Tailwind Classes |
|-------|-----------------|
| Unselected | `bg-white text-gray-600 border border-gray-200 hover:border-gray-300` |
| Selected | `bg-gray-900 text-white` |
| Disabled | Not applicable — all chips always enabled; filtering is client-side |

"All …" chip serves as the clear affordance. No separate "Clear" button is
rendered (`showClear={false}`). When vertical=all AND type=all AND status=all,
no clear affordance is needed.

### Filtering Behavior

Filtering is **client-side** against the already-fetched `drafts` array. No
additional server call on chip change. Three filter states are composed with
AND logic:
- `vertical='all'` passes all; otherwise match `draft.vertical === verticalFilter`
  or (`verticalFilter === 'cross'` AND `draft.vertical === null`)
- `type='all'` passes all; otherwise match `draft.metadata?.template_type === typeFilter`
- `status='all'` passes all; otherwise match `draft.status === statusFilter`

Filtered count is shown inline in the Pending tab badge:
`Pending ({filteredCount})` when any filter is non-`all`.

## Surface 3: Pending Tab — Bulk-Action Toolbar

### Visibility Rule

The toolbar is hidden (`display: none` / conditional render) when
`selectedIds.size === 0`. It appears (`display: flex`) when
`selectedIds.size >= 1`. No animation — instant show/hide via conditional JSX.

### Toolbar Layout

```
<div className="flex items-start gap-3 px-4 py-3 bg-gray-50 border border-gray-200 rounded-lg mb-3">
  [Shared reject reason textarea — left, flex-1]
  [Action buttons — right, flex-shrink-0, flex-col gap-2]
</div>
```

The `<textarea>` for shared reject reason is rendered LEFT of the action
buttons (per D-06: "rendered above the bulk-action buttons"). Adjust: for
horizontal toolbar layout, reason field is on the left; buttons on the right.
The reason field has `placeholder="Optional reason for rejecting all selected drafts…"`,
3 rows, uses the `Textarea` component from `FormElements.jsx` with
`className="text-sm"`. The field is always visible when the toolbar is open
(not toggled). Its value is passed as `reason` to `bulkRejectDrafts` when
Reject is triggered.

### Buttons

Two buttons, stacked vertically on the right side of the toolbar:

```
<Button variant="primary" size="sm" onClick={handleBulkApprove}>
  <Check size={14} className="mr-1" />
  Approve selected ({selectedIds.size})
</Button>

<Button variant="danger" size="sm" onClick={handleBulkReject}>
  <X size={14} className="mr-1" />
  Reject selected ({selectedIds.size})
</Button>
```

**Toolbar button states:**

| State | Approve button | Reject button |
|-------|----------------|---------------|
| Idle, ≥1 selected | `variant="primary"` enabled | `variant="danger"` enabled |
| Bulk operation in flight | `disabled` + `loading` prop | `disabled` |
| Post-confirm (modal open) | Not applicable — modal takes over | Not applicable |

The approve button uses `bg-brand-500` (orange) per the design-system `primary`
variant. The reject button uses `bg-red-600` per the `danger` variant.

### Empty-to-Active Transition

No animation. Toolbar mounts when `selectedIds.size` transitions from 0 to 1;
unmounts when it returns to 0. This keeps the component tree simple and avoids
retained textarea state after deselection.

## Surface 4: Bulk Confirmation Modal

### Trigger

Both "Approve selected (N)" and "Reject selected (N)" toolbar buttons open this
modal. The modal is controlled by `bulkConfirmState` in component state:
```
null | { action: 'approve' | 'reject', draftIds: string[], phase: 'confirm' | 'executing' | 'done' }
```

### Modal Size

`size="lg"` (`max-w-2xl`). The execution feed may contain up to ~120 rows; the
modal body scrolls independently.

### Pre-Confirm Phase (`phase='confirm'`)

**ModalHeader:**
```
<ModalTitle>
  {action === 'approve' ? 'Approve' : 'Reject'} {draftIds.length} draft{draftIds.length !== 1 ? 's' : ''}?
</ModalTitle>
```

**ModalContent:**
A `<ul className="space-y-1 mb-4">` listing the first 5 draft slugs/names:
```html
<li className="text-sm text-gray-700">• {draft.name or draft.id slice(0,8)}</li>
```
Followed by, if `draftIds.length > 5`:
```html
<p className="text-sm text-gray-500">…and {draftIds.length - 5} more</p>
```

For Reject action: the shared `reason` value (from the toolbar textarea) is
echoed back as read-only hint text below the name list:
```html
<p className="text-sm text-gray-500 italic mt-2">
  Reason: "{reason}" (applied to all)
</p>
```
If reason is empty: no echo line rendered.

**ModalFooter:**
```
<Button variant="secondary" onClick={closeModal}>Keep reviewing</Button>
<Button
  variant={action === 'approve' ? 'primary' : 'danger'}
  onClick={handleConfirm}
>
  Confirm {action === 'approve' ? 'approve' : 'reject'}
</Button>
```

### Executing Phase (`phase='executing'`)

Once Confirm is clicked:
1. Modal title updates to `"Processing {draftIds.length} drafts…"`.
2. Cancel and Confirm buttons are replaced by a single disabled
   `<Button variant="secondary" disabled>Please wait…</Button>`.
3. ModalContent renders a scrollable execution feed:

```html
<div
  data-testid="bulk-exec-feed"
  className="max-h-80 overflow-y-auto space-y-1 font-mono text-xs border border-gray-100 rounded bg-gray-50 p-3"
>
  {execResults.map((r) =>
    r.ok
      ? <p className="text-green-700">✓ {r.slug}</p>
      : <p className="text-red-600">✗ {r.slug} — {r.error}</p>
  )}
</div>
```

Each result row appends as the bulk EF call returns per-draft status. The
container auto-scrolls to the bottom as new rows append (use `useEffect` on
`execResults.length` to scroll the div's `scrollTop` to `scrollHeight`).

**Error row format:** `✗ <slug> — <error message>`

The error message is the raw `error` string from the EF result. Truncate at 120
characters with `…` to prevent overflow.

### Done Phase (`phase='done'`)

Once all IDs are processed:
1. Final summary line appended to the feed (styled distinctly):

```html
<p className="mt-2 pt-2 border-t border-gray-200 text-sm font-semibold text-gray-900">
  {action === 'approve' ? 'Approved' : 'Rejected'} {successCount},
  {failCount > 0 ? ` ${failCount} failed (see errors above)` : ' all succeeded'}
</p>
```

2. The "Please wait…" button is replaced by:
```html
<Button variant="secondary" onClick={closeModalAndReload}>Close summary</Button>
```

3. `closeModalAndReload` calls `loadDrafts()` after closing so the Pending
   list refreshes — failed drafts remain visible with `pending` status.

### Modal Close Behavior

- Pre-confirm: Escape key closes modal (design system Modal default).
- Executing: `closeOnEscape={false}` and `closeOnOverlay={false}` — admin must
  wait for the operation to complete or reach the done state.
- Done: Escape key re-enabled; overlay click closes.

## Surface 6: Generate Tab — Vertical-Scoped Template Type Dropdown

### Behavior

When `vertical` changes, the `TEMPLATE_TYPE_OPTIONS` list passed to the
`<Select>` narrows to types relevant to that vertical. The current selection
is reset to the first option of the filtered list when it is no longer in the
filtered set.

**Mapping table (locked by D-12):**

```js
const VERTICAL_TYPE_MAP = {
  'cross-vertical':  ['menu', 'promo', 'announcement', 'reminder', 'wayfinding', 'health_tip', 'queue_status', 'drive_thru', 'waiting_room_ambient', 'emergency_alert'],
  'restaurants':     ['menu', 'promo', 'announcement', 'reminder', 'wayfinding', 'drive_thru', 'queue_status'],
  'retail':          ['promo', 'announcement', 'reminder', 'wayfinding', 'health_tip'],
  'healthcare':      ['announcement', 'reminder', 'health_tip', 'waiting_room_ambient', 'queue_status', 'emergency_alert'],
};
```

`cross-vertical` shows all 10 types. When admin selects a specific vertical,
only that vertical's types are in the `<select>` option list.

**UX pattern: filtered list (not disabled options).** Options not in the
filtered set are simply absent from the `<select>` element — they are not
rendered as `disabled` `<option>` elements. This avoids confusion about
greyed-out choices. Implementation: derive `filteredTypeOptions` from
`VERTICAL_TYPE_MAP[vertical]`, then filter `TEMPLATE_TYPE_OPTIONS` to only
include matching entries.

**Reset on vertical change:** In the `setVertical` handler, if `templateType`
is not in the new filtered set, reset `templateType` to the first value of
the filtered list.

**States:**

| State | Visual |
|-------|--------|
| Cross-vertical selected | All 10 options visible |
| Specific vertical selected | Subset of options; no disabled options |
| Submitting | Select is `disabled` (all form fields disabled) |

## Accessibility Anchors

These are required; the checker will verify presence.

- Filter chip rows: each chip group `<div>` has `role="group"` and
  `aria-label="Filter by {vertical/type/status}"`.
- Header checkbox: `aria-label="Select all visible drafts"`.
- Per-row checkbox: `aria-label="Select draft {slug}"`.
- Bulk toolbar: `role="toolbar"` `aria-label="Bulk actions"`. Appears only
  when ≥1 draft is selected; toolbar receives focus on first appearance
  (optional enhancement, not blocking).
- Bulk confirm modal: `aria-labelledby` pointing to `ModalTitle` id (design
  system Modal default handles this via `aria-modal` + focus trap).
- Execution feed: `aria-live="polite"` on the feed container so screen readers
  announce appended rows.
- Orientation select: `<label htmlFor="gen-orientation">` wired to `id`.
- Template type select (Phase 178 extension): no new aria needed — it is the
  same `<Select>` with fewer options; existing `data-testid="gen-type"` is
  unchanged.
