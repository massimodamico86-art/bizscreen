Wave 7 (gap closure) — Close BL-03 (reject lacks already-rejected idempotency
+ races against approve). Two concrete failure modes verified in production code:

1. **No idempotency on already-rejected drafts.** Handler refuses approved
   drafts (lines 32-34) but happily re-rejects an already-rejected draft,
   OVERWRITING the original reviewer's reviewed_by/reviewed_at/rejected_reason.
   Audit trail destroyed — original rejector silently replaced. The approve
   handler at lines 111-118 has the symmetric guard for already-approved; reject
   is missing the symmetry.

2. **No .eq('status', 'pending') race guard on UPDATE.** If admin A's approve
   and admin B's reject fire concurrently, both read status='pending'. Approve
   completes first (svg_templates row inserted). Reject's UPDATE then fires
   with no WHERE-clause guard and flips the JUST-APPROVED draft to 'rejected'
   while the published svg_templates row remains active. Result: live published
   template tied to a 'rejected' draft. Audit-integrity violated.

to the approve handler's existing idempotency check.

NEW reject.ts shape (after BL-03 closure):
```typescript
export async function reject(body, supabase): Promise<RejectResult> {
  const { data: draft, error: fetchErr } = await supabase
    .from("template_drafts")
    .select("metadata, status")
    .eq("id", body.draftId)
    .single();
  if (fetchErr || !draft) throw new Error(`Draft not found: ${body.draftId}`);

  // BL-03 closure 177-09: idempotency guard symmetric to approve.ts:111-118.
  // Re-rejecting must not overwrite the original reviewer's audit metadata.
  if (draft.status === "rejected") {
    return { ok: true, draftId: body.draftId };
  }

  // Pre-existing T-177-11 mitigation: refuses to flip an already-approved draft.
  if (draft.status === "approved") {
    throw new Error("Cannot reject an already-approved draft");
  }

  // BL-03 closure 177-09: race-guard WHERE clause — only flip if STILL pending.
  // If admin B's reject fires after admin A's approve completes, the .eq('status','pending')
  // WHERE clause matches 0 rows → PGRST116 from PostgREST .single() → throw documented race error.
  // This pairs with the approve handler's RPC-side advisory lock (177-08) to make
  // the approve-vs-reject race deterministic instead of last-write-wins.
  const { data, error } = await supabase
    .from("template_drafts")
    .update({
      status: "rejected",
      metadata: {
        ...(draft.metadata ?? {}),
        reviewed_by: body.reviewerUid,
        reviewed_at: new Date().toISOString(),
        rejected_reason: body.reason ?? null,
      },
    })
    .eq("id", body.draftId)
    .eq("status", "pending")        // ← BL-03 race guard
    .select("id")
    .single();

  if (error) {
    // PostgREST returns code 'PGRST116' when .single() finds zero rows.
    // This means the WHERE clause didn't match — draft was concurrently approved
    // or already rejected (but we already checked status above, so this is the
    // approve-race case).
    if (error.code === "PGRST116" || /no.*rows/i.test(error.message ?? "")) {
      throw new Error("Cannot reject — draft was concurrently approved or already rejected");
    }
    throw error;
  }

  return { ok: true, draftId: data.id };
}
```
