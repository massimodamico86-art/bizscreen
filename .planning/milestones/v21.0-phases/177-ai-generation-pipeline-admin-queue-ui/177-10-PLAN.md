<objective>
Wave 7 (gap closure) — Close BL-01 (`saveDraftSvgContent` does direct UPDATE
bypassing EF re-validation). The Wave 4 inline edit modal calls
`saveDraftSvgContent(draft.id, editedSvg)` directly against the JWT-scoped
supabase client — `UPDATE template_drafts SET svg_content = ?`. The plan banks
on RLS `template_drafts_admin_only` to gate the UPDATE; that prevents non-admins
from writing but does NOT prevent an admin from writing malformed/XSS svg_content
into a pending draft (which is then visible in the Pending tab via
dangerouslySetInnerHTML, AND — combined with BL-04 — exfiltrable to the
post-approval public TemplateGalleryPage).

Client-side validateSvg in TemplateDraftEditModal is feedback-only; it can be
bypassed via devtools. Server-side re-validation is missing on the SAVE path
(it exists on the APPROVE path at handlers/approve.ts:143, but only catches
content that comes through the approve flow — admin can save malformed SVG to
a pending draft and walk away, leaving it visible to other admins viewing the
queue).

Purpose: Move the save_edit operation server-side so the same `validateSvg` gate
that protects the ingest path (handlers/generate.ts) and the approve path
(handlers/approve.ts:143) ALSO protects the admin-edit path. Defense-in-depth
restored — no svg_content writes to template_drafts can skip server-side
validation.

Why depends on 177-07: this plan creates a new EF handler that calls the
same `validateSvg` from svgValidator.ts. The 177-07 hardened validator (with
@import / url(http(s):) / url(//) / javascript: blocks) MUST land BEFORE this
handler ships, otherwise the new server-side gate has the same gap as the old
client-side gate.

Output: New `handlers/saveEdit.ts`, new dispatch branch in `index.ts`,
refactored `saveDraftSvgContent` in the frontend service, vitest spec asserting
the wiring + the validate-before-update source order, and a [BLOCKING] EF
redeploy task to push the new handler live.

    ```typescript
    //
    // Server-side re-validation gate for the Wave 4 inline edit modal's Save path.
    // Mirrors the B1 re-validation pattern from handlers/approve.ts:120-156:
    //   1. Fetch draft (refuse if not 'pending' or 'needs_human_review')
    //   2. Run validateSvg with deno-dom DOMParser + DOMPurify=null
    //      (RESEARCH §Constraint 4 — Rule 4 silently skipped server-side)
    //   3. On pass: UPDATE template_drafts.svg_content via service-role client
    //      (RLS double-defense per D-18 — admin gate at index.ts:53-55 +
    //      template_drafts_admin_only RLS policy from Phase 176)
    //   4. On fail: throw 422 Response with { error, issues }
    //
    // Defense-in-depth: client-side validateSvg in TemplateDraftEditModal.jsx is
    // feedback-only and devtools-bypassable. The server-side gate here is the
    // load-bearing security boundary.
    //
    // Inherits 177-07 hardening — the FORBIDDEN_CONTENT_TOKENS in
    // src/services/svgValidator.js (re-exported via svgValidator.ts) now blocks
    // @import / url(http(s):) / url(//) / javascript: pseudo-protocols, so this
    // handler's validateSvg gate catches CSS-injection payloads on the
    // admin-edit path AS WELL AS the ingest path (handlers/generate.ts) and
    // the approve-time re-validation (handlers/approve.ts:143).
    //
    // Source-order awk gate (verified by tests/integration/saveEditValidation.test.js Test 5):
    //   validateSvg(body.svgContent → .from("template_drafts").update(

    import { validateSvg } from "../svgValidator.ts";

    interface SaveEditBody {
      draftId: string;
      svgContent: string;
      callerUid: string;
    }

    interface SaveEditResult {
      ok: true;
      draftId: string;
    }

    export async function saveEdit(
      body: SaveEditBody,
      // deno-lint-ignore no-explicit-any
      supabase: any,
    ): Promise<SaveEditResult> {
      // Step 1 — fetch + validate status.
      const { data: draft, error: fetchErr } = await supabase
        .from("template_drafts")
        .select("status")
        .eq("id", body.draftId)
        .single();
      if (fetchErr || !draft) throw new Error(`Draft not found: ${body.draftId}`);
      if (draft.status !== "pending" && draft.status !== "needs_human_review") {
        throw new Error(
          `Cannot save edit on draft in status '${draft.status}' — only pending/needs_human_review drafts can be edited`,
        );
      }

      // Step 2 — RE-VALIDATE the edited svg_content server-side (BL-01 closure).
      // Same DOMParser + DOMPurify config as handlers/approve.ts:120-146.
      const denoDomSpecifier = "jsr:" + "@b-fuze/deno-dom";
      const denoDomMod = await import(/* @vite-ignore */ denoDomSpecifier);
      const DenoDOMParser = denoDomMod.DOMParser;
      class DenoSvgDOMParser {
        // deno-lint-ignore no-explicit-any
        parseFromString(input: string, _mime: string): any {
          return new DenoDOMParser().parseFromString(input, "text/html");
        }
      }

      const validatorResult = validateSvg(body.svgContent, {
        DOMParserCtor: DenoSvgDOMParser,
        DOMPurify: null,
      });
      if (!validatorResult.ok) {
        // 422 — semantically unprocessable; draft.svg_content unchanged.
        throw new Response(
          JSON.stringify({
            error: "Edit failed server-side validation",
            issues: validatorResult.errors,
          }),
          { status: 422, headers: { "Content-Type": "application/json" } },
        );
      }

      // Step 3 — UPDATE template_drafts.svg_content (only after validator passed).
      const { error: updateErr } = await supabase
        .from("template_drafts")
        .update({ svg_content: body.svgContent })
        .eq("id", body.draftId);
      if (updateErr) throw updateErr;

      return { ok: true, draftId: body.draftId };
    }
    ```

**Attacker model:** Compromised admin (or admin with malicious intent) opens
the Edit modal, opens devtools, modifies the modal's local state to bypass the
client-side validateSvg gate, and clicks Save & Publish. Pre-fix:
saveDraftSvgContent does direct UPDATE → svg_content with @import payload lands
in template_drafts → other admins viewing the queue trigger DNS prefetch on
the attacker's domain → after approval, content lands in svg_templates → ALL
TENANTS viewing TemplateGalleryPage trigger the same prefetch.

**Mitigation:** EF action=save_edit runs validateSvg with the 177-07 hardened
rules → @import is rejected → 422 returned to client → svg_content unchanged
in template_drafts → no admin or tenant ever sees the malicious content.

can't — the EF is the only write path now), the approve handler's B1
re-validation (handlers/approve.ts:143) provides a second gate.
