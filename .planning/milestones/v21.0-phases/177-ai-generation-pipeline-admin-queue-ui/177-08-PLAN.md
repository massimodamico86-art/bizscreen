<objective>
Wave 6 (gap closure) — Close BL-02 (S3-then-INSERT non-atomic + INSERT-then-UPDATE
non-atomic), BL-06 (read-then-act idempotency race / no row lock), and WR-09
(Date.now() in S3 key produces orphan accretion on retry). Combined, these
violate TADM-03's atomicity contract — the phase-goal language "failure of any
step leaves draft unchanged (no partial publish)" is structurally false in
production code today.

Purpose: Restore TADM-03's "approve is atomic" contract by:
1. Wrapping `INSERT svg_templates` + `UPDATE template_drafts` in a single
   Postgres transaction via a new SQL function `approve_draft_atomic` (BL-02 fix).
2. Taking a row-level advisory lock keyed on `draftId` at the start of the
   function so concurrent invocations serialize (BL-06 fix). Second concurrent
   call sees the lock fail and short-circuits.
3. Folding the idempotency short-circuit (status='approved' → return cached
   metadata) inside the transaction so it's race-free (BL-06 fix complement).
4. Switching the S3 key from `${slug}-${Date.now()}.png` to deterministic
   `${slug}.png` (i.e. `ai-${draft.id}.png`) so retries overwrite the same
   object (WR-09 fix folded in because it ships in the same surface area).

Output: New SQL migration `177_approve_draft_atomic.sql`, refactored approve
handler that calls the RPC, deterministic S3 key, atomicity test (pure-mock),
and a [BLOCKING] `supabase db push` task to apply the migration to the live DB
before EF redeploy.

```sql
CREATE OR REPLACE FUNCTION approve_draft_atomic(
  p_draft_id        UUID,
  p_svg_template    JSONB,    -- full INSERT-row payload (name, description, category, orientation, thumbnail, svg_url, svg_content, width, height, tags, is_active, vertical, metadata)
  p_metadata_patch  JSONB     -- patch to merge into template_drafts.metadata: { reviewed_by, reviewed_at, thumbnail_url, svg_template_id }
) RETURNS JSONB                -- { ok: true, svg_template_id: uuid, thumbnail_url: text } OR raises 'concurrent_approve' / 'draft_not_found' / 'already_approved'
LANGUAGE plpgsql
SECURITY DEFINER               -- SAFE: callable only via Edge Function service-role; not exposed via PostgREST default schema
SET search_path = public
AS $$
DECLARE
  v_existing_template_id UUID;
  v_thumbnail_url        TEXT;
  v_new_template_id      UUID;
  v_draft_status         TEXT;
BEGIN
  -- Row-level advisory lock — serializes concurrent approves on same draft.
  IF NOT pg_try_advisory_xact_lock(hashtext(p_draft_id::text)) THEN
    RAISE EXCEPTION 'concurrent_approve_in_progress';
  END IF;

  -- Fetch current draft.status FOR UPDATE.
  SELECT status, metadata->>'svg_template_id', metadata->>'thumbnail_url'
    INTO v_draft_status, v_existing_template_id, v_thumbnail_url
    FROM template_drafts
    WHERE id = p_draft_id
    FOR UPDATE;

  IF NOT FOUND THEN
    RAISE EXCEPTION 'draft_not_found';
  END IF;

  -- Idempotency: already-approved → return cached metadata.
  IF v_draft_status = 'approved' THEN
    RETURN jsonb_build_object(
      'ok', true,
      'svg_template_id', v_existing_template_id,
      'thumbnail_url', v_thumbnail_url,
      'idempotent', true
    );
  END IF;

  -- INSERT svg_templates from the JSONB payload.
  INSERT INTO svg_templates (
    name, description, category, orientation, thumbnail, svg_url,
    svg_content, width, height, tags, is_active, vertical, metadata
  ) VALUES (
    p_svg_template->>'name',
    p_svg_template->>'description',
    p_svg_template->>'category',
    p_svg_template->>'orientation',
    p_svg_template->>'thumbnail',
    p_svg_template->>'svg_url',
    p_svg_template->>'svg_content',
    (p_svg_template->>'width')::INT,
    (p_svg_template->>'height')::INT,
    ARRAY(SELECT jsonb_array_elements_text(COALESCE(p_svg_template->'tags', '[]'::jsonb))),  -- jsonb array → text[] (matches Phase 173 migration's array-construction pattern; per checker WARNING 1 — use this exact form, no further patching)
    (p_svg_template->>'is_active')::BOOLEAN,
    p_svg_template->>'vertical',
    p_svg_template->'metadata'
  ) RETURNING id INTO v_new_template_id;

  -- UPDATE draft (LAST step, in same transaction as the INSERT).
  UPDATE template_drafts
    SET status = 'approved',
        metadata = COALESCE(metadata, '{}'::jsonb) || p_metadata_patch ||
                   jsonb_build_object('svg_template_id', v_new_template_id)
    WHERE id = p_draft_id
      AND status IN ('pending', 'needs_human_review');   -- defensive WHERE — also catches racing reject

  IF NOT FOUND THEN
    -- Should never happen because of FOR UPDATE above, but defensive.
    RAISE EXCEPTION 'draft_status_changed_during_approve';
  END IF;

  RETURN jsonb_build_object(
    'ok', true,
    'svg_template_id', v_new_template_id,
    'thumbnail_url', p_metadata_patch->>'thumbnail_url'
  );
END;
$$;
```

**Attacker model:**
1. (BL-02) Admin clicks Approve. EF rasterizes → S3 PUT succeeds → INSERT svg_templates fails (transient DB hiccup). Pre-fix: PNG orphaned in S3; admin retries; new S3 key (Date.now()); retry succeeds; svg_templates has 1 row but S3 has 2 PNGs (1 orphan).
2. (BL-06) Admin double-clicks Approve. Pre-fix: 2 concurrent EF invocations both read draft.status='pending'; both rasterize; both PUT to S3 (different keys); both INSERT svg_templates (no unique constraint); both UPDATE draft. Result: 2 svg_templates rows for 1 draft, 2 S3 PNGs.
3. (WR-09) Each retry of (1) accretes another orphan PNG indefinitely.

**Mitigation:** The RPC's transaction + advisory lock + deterministic S3 key
collapses all three threats into a single closure. The EF retry of (1) now
runs through the same RPC; if the RPC's INSERT-then-UPDATE transaction had
failed and rolled back, the retry sees no svg_templates row → INSERT succeeds.
The deterministic S3 key means the rasterize+upload of the retry overwrites
the previous PNG.

**Residual risk:** T-177-20a (concurrent-approve error message leakage to admins)
