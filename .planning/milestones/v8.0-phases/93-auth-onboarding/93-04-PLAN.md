<context>
**Root Cause Analysis:**

`VITE_DEV_BYPASS_AUTH=true` is set in `.env.local`. This is a Vite compile-time environment variable that gets baked into the app bundle at build/serve time via `import.meta.env.VITE_DEV_BYPASS_AUTH`. When true:

1. `AuthContext` (src/contexts/AuthContext.jsx line 10): Sets `DEV_AUTH_BYPASS = true`, providing a mock authenticated user immediately with `has_completed_onboarding: true`
2. `LoginPage` (src/auth/LoginPage.jsx line 33-37): `useEffect` auto-redirects to `/app` before the form renders
3. `PublicRoute` (src/router/AppRouter.jsx line 67-78): `Navigate` redirects `/auth/login` and `/auth/signup` to `/app` because `user` is truthy

The fix must happen at the Vite server level, not at the Playwright storage/cookie level, because `VITE_DEV_BYPASS_AUTH` is evaluated at JavaScript module load time.

**Solution:** Add `env: { VITE_DEV_BYPASS_AUTH: 'false' }` to the `webServer` config in `playwright.config.js`. Playwright's `webServer.env` merges with the process environment and Vite picks up env vars from both `.env.local` AND `process.env`. However, Vite's `import.meta.env.VITE_*` vars are sourced from `.env*` files AND the `define` option, but NOT from `process.env` directly.

**Correct approach:** Since Vite loads `VITE_*` from `.env.local` at dev server startup, we need to either:
1. Use `webServer.command` to pass the env var explicitly: `VITE_DEV_BYPASS_AUTH=false npm run dev`
2. Or use a `.env.test.local` file that overrides `.env.local` for test runs

The simplest and most reliable approach is option 1: prefix the webServer command with the env var override. On macOS/Linux, `VITE_DEV_BYPASS_AUTH=false npm run dev` will set the var in the process environment, and Vite reads `VITE_*` from `process.env` as well (since Vite 5+, env vars prefixed with VITE_ in process.env are exposed to client code).

**Important constraint:** The `reuseExistingServer: true` setting means if a dev server is already running (with bypass enabled), Playwright will use it. The tests must either:
- Ensure no dev server is running when tests start (CI behavior), OR
- Accept that `reuseExistingServer` may pick up an existing bypass-enabled server

For this gap closure, we will modify the webServer command AND add a note that for screenshot-producing test runs, the existing dev server should be stopped first. Alternatively, we set `reuseExistingServer: !process.env.CI` to always start fresh in CI.

The `webServer.env` option allows setting environment variables for the spawned server process.

From Vite docs: Environment variables prefixed with VITE_ are exposed to client code. Vite reads them from:
1. `.env` files (.env, .env.local, .env.[mode], .env.[mode].local)
2. `process.env` (Node.js environment)

So setting `VITE_DEV_BYPASS_AUTH=false` in the process environment for the webServer command will override the `.env.local` value.
