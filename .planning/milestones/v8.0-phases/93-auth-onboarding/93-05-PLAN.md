**Root Cause:** The `screenshots/onboarding/` directory does not exist because the onboarding tests were never run against a live dev server. The test file is correctly written and appears in `playwright --list`, but no screenshots were ever produced.

**Expected behavior with dev bypass disabled (from 93-04):**
When `VITE_DEV_BYPASS_AUTH=false` is set via the Playwright webServer command (from plan 93-04), Supabase auth is no longer bypassed. However, the Supabase local instance (127.0.0.1:54321) may not be running in the test environment.

**What happens when tests run:**
1. The `auth.setup.js` attempts to authenticate via the login form
2. Without Supabase running, the login may fail or the dev bypass mock user won't be available
3. The onboarding tests use `loginAndPrepare(page)` which depends on a valid auth session

**The onboarding test flow with dev bypass active (VITE_DEV_BYPASS_AUTH=true scenario):**
- The mock user has `has_completed_onboarding: true` in the DEV_MOCK_PROFILE
- The onboarding wizard will NOT appear (it's gated by `current_unified_step !== 'complete'`)
- The tests are designed to handle this with fallback screenshots:
  - AUTH-10: Captures `01-dashboard-onboarding-complete` (dashboard showing onboarding already done)
  - AUTH-11: Tries settings, then captures `03-settings-no-industry-picker` or `03-dashboard-no-industry-access`
  - AUTH-12: Navigates to `/app/screens`, tries to find a pair button, captures whatever state is visible

**Key insight:** The onboarding tests need the dev bypass to be active (or real Supabase auth) for `loginAndPrepare` to work. With 93-04's change disabling the bypass, the auth setup may fail without Supabase. There are two approaches:

**Approach A (preferred):** The onboarding tests should work regardless of bypass state. Since the test's `beforeEach` skips when `testInfo.project.name !== 'chromium'` and when `!process.env.TEST_USER_EMAIL`, and the chromium project depends on the setup project which saves auth state, the tests should work IF the auth setup succeeds.

With `VITE_DEV_BYPASS_AUTH=false` in the webServer command:
- The dev server starts without bypass
- `auth.setup.js` navigates to `/auth/login` -- the login form renders
- Without Supabase, filling credentials and clicking submit will show an error
- `authenticateRole` will fail to reach `/app`
- The chromium project's storageState may be empty/invalid

**This means we might need to handle both scenarios.** The safest approach:

1. Check if `reuseExistingServer` picks up an already-running dev server with bypass enabled
2. If a dev server IS running with bypass, the onboarding tests can use the mock auth
3. If no dev server is running, Playwright starts one WITHOUT bypass, and onboarding tests need real Supabase

**Practical solution for the gap closure:**
- The onboarding tests already skip gracefully when `!process.env.TEST_USER_EMAIL`
- When the webServer starts with bypass disabled AND no Supabase, auth setup fails silently (saves empty state)
- The onboarding test's `loginAndPrepare` may navigate to `/app` but without auth, redirects to login
- The test's fallback logic should still capture SOMETHING, but might hit the login page instead of dashboard

**Most reliable approach for this gap closure:**
Run the onboarding tests in a separate command that temporarily sets `VITE_DEV_BYPASS_AUTH=true` for the webServer, ensuring the mock auth works and the tests can reach the dashboard. The onboarding wizard won't appear (mock user has completed onboarding), but the fallback screenshots WILL be captured.

We can do this by running: `VITE_DEV_BYPASS_AUTH=true npx playwright test tests/e2e/onboarding-wizard-screenshots.spec.js --project=chromium`

This ensures:
- Dev server starts with bypass enabled (if reuseExistingServer picks up one, or Playwright starts a new one)
- Auth setup succeeds with the mock user
- Onboarding tests reach the dashboard
- Fallback screenshots are captured in screenshots/onboarding/

**But wait** -- with 93-04's change, the webServer command is now `VITE_DEV_BYPASS_AUTH=false npm run dev`. If we run the above, the VITE_DEV_BYPASS_AUTH=true in the outer process won't affect the webServer's spawned process because Playwright explicitly sets its own command.

**Resolution:** For these onboarding tests, we don't need to change the config. Instead:
1. Ensure a dev server IS running with bypass enabled (the default `.env.local` state) before running onboarding tests
2. Since `reuseExistingServer: true` (locally), Playwright will use it
3. OR: Run the tests with the server already started manually

**Alternative (cleaner):** Modify the webServer config from 93-04 to use an environment variable to control the override:

```javascript
command: `VITE_DEV_BYPASS_AUTH=${process.env.KEEP_DEV_BYPASS === 'true' ? 'true' : 'false'} npm run dev`
```

This adds unnecessary complexity. Let's keep it simple:

**Final approach:** The simplest fix is to just run the onboarding tests NOW, before 93-04's config change takes effect. The dev server with bypass is already the default. OR: run them specifically with `reuseExistingServer: true` while a bypass-enabled server is running.

Actually, looking at this more carefully, the cleanest approach is:
1. The executor should start a dev server with bypass enabled: `npm run dev`
2. Then run: `npx playwright test tests/e2e/onboarding-wizard-screenshots.spec.js --project=chromium`

The test will:
- Auth setup: navigates to `/auth/login`, gets redirected to `/app` (bypass active), saves that state
- Onboarding tests: navigate to `/app`, find dashboard (onboarding completed for mock user), capture fallback screenshots
