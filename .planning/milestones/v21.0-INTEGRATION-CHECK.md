### WARNING — 4

#### WARNING-1 (CARRIED): UI auth gate gap for AdminTemplateQueuePage

**Affects:** TADM-04
**Connection:** App.jsx `shouldShowClientUI` OR-logic → AdminTemplateQueuePage render
**Status:** PARTIAL (data protected; UI render not architecturally gated)

`App.jsx:705-708` defines `shouldShowClientUI = role==='client' || isImpersonating || isAdminToolPage`. Because `admin-template-queue` is in the `adminToolPages` allowlist (L701) and the conditions are OR'd, a hypothetical client-role user whose `currentPage` is `admin-template-queue` resolves `shouldShowClientUI=true` and enters the client UI branch which renders `pages['admin-template-queue']` = AdminTemplateQueuePage. Data access is blocked by RLS + EF (returns empty rows, not 403), but the page component mounts and `fetchPendingDrafts` fires.

**Why this is no longer BLOCKER-equivalent:** BLOCKER-1 closure now gates the tile inside the super-admin-only `SuperAdminDashboardPage` render path, so there is no production navigation path for a client to set this `currentPage`. The architectural gap remains as a defense-in-depth concern.

**Fix (low priority):** Add a role check in the client-UI render path; if `isAdminToolPage && authUserProfile?.role === 'client'`, redirect to dashboard or 403.

#### WARNING-2 (CARRIED): TVRZ-02 perf budget — ACCEPTED-FOR-v21.0

**Affects:** TVRZ-02, TVRZ-04
**Connection:** Phase 179 SC-2 perf gate ↔ live-cloud + 485-template configuration
**Status:** PARTIAL (codebase WIRED; runtime budget unmet, accepted)

Plan 180-10 prod-build re-run measured `[SC-2] gallery first-paint: 9753ms` against `npm run build && npx vite preview --port 4173` with cloud Supabase (`gdxizdiltfqeugbsgtpx`, 485 templates). The 1000ms SC-2 budget was set in Phase 179 CONTEXT against an unmeasured assumption; empirical reality is ~9.7s. Dev-mode-bundle hypothesis falsified (only 6.5% reduction from dev-mode 10434ms). SC-9 (skeleton-flash) is structurally unreachable until SC-7 first-paint completes within 5s precondition — so SC-9 deferred together.

**Disposition:** option-defer per Plan 180-10 Task 4; ACCEPTED-FOR-v21.0; recorded in deferred-items.md Phase 180 acceptance section (items 7 + 8). Real perf remediation deferred to v21.1.

#### WARNING-3 (CARRIED, INFORMATIONAL): AI-approved templates have slug=NULL

**Affects:** TADM-03, TCAT-01 (informational only — no functional regression)
**Connection:** approve.ts `p_svg_template` payload → approve_draft_atomic INSERT → gallery_templates VIEW
**Status:** WIRED (schema-legal NULL)

`handlers/approve.ts:190-210` constructs `p_svg_template` JSONB payload without a `slug` field. `svg_templates.slug` is nullable (migration 167:21) with a partial unique index `WHERE slug IS NOT NULL`. AI-generated approved templates appear in `gallery_templates` with `slug = NULL` — addressable only by `id`. If any downstream feature attempts `WHERE slug = 'x'` to find an AI-generated template, it returns no rows silently. Not a regression; tech-debt item already tracked in v21.0-MILESTONE-AUDIT.md tech_debt.

#### WARNING-4 (CARRIED, INFORMATIONAL): verify-178-counts.cjs `.catch()` on PostgrestBuilder crashes

**Affects:** TCAT-04 (verification-harness reliability only — verdicts independently re-verified)
**Connection:** Phase 178 verification automation → SC-gate harness
**Status:** WIRED but FRAGILE

`scripts/verify-178-counts.cjs:174-176` still chains `.catch()` on `supabase.rpc('exec_sql', ...)` — that returns a PostgrestBuilder (a thenable, not a real Promise), so `.catch()` does not behave as a rejection handler. 178-VERIFIER-AUDIT.md F-01 documented that the script crashes at the TCAT-04 check; the underlying TCAT-04 verdict was independently re-verified by the auditor. Phase 180 did not fix this script; v21.0-MILESTONE-AUDIT.md tech_debt also notes the harness itself is broken.

**Fix:** Replace `.catch()` chain with `try { const { data, error } = await supabase.rpc(...); } catch (e) { ... }`. Non-functional fix.
