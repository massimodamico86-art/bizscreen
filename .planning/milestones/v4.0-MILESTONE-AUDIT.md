## Cross-Phase Integration

### Integration Findings

#### INT-01 (High) — Duplicate Heartbeat RPCs

**From:** `usePlayerContent.sendHeartbeat()` (legacy)
**To:** `usePlayerHeartbeat.updateDeviceStatus()` (v4.0)
**Affected:** TELM-01, ALRT-01, ALRT-02

`usePlayerContent` (line 298) still calls the legacy `player_heartbeat` RPC on its 30-second content polling loop. This runs unsynchronized alongside `usePlayerHeartbeat`'s `update_device_status` call. Both update `last_seen_at`, causing unnecessary DB writes and theoretically could mask offline detection if only the full heartbeat pipeline fails.

**Remediation:** Remove `sendHeartbeat()` call from `usePlayerContent` polling loop. The `usePlayerHeartbeat` hook now owns all device status reporting.

#### INT-02 (Medium) — Wrong `lastActivityRef` Passed to Stuck Detection

**From:** `usePlayerPlayback.lastActivityRef` (video frame progress)
**To:** `useStuckDetection` via ViewPage
**Affected:** RECV-01

ViewPage passes `lastActivityRef` from `usePlayerContent` to `useStuckDetection`, but the content-level ref is only updated on content load/reconnect, not on video frame advances. The `usePlayerPlayback` hook maintains its own `lastActivityRef` that updates on every video `timeupdate`, but ViewPage does not destructure it.

**Remediation:** Destructure `lastActivityRef` from `usePlayerPlayback` return value and pass it to `useStuckDetection`.

#### INT-03 (Low/Design) — Content Verification Playlist-Only Transitions

**Affected:** VERI-03, VERI-04

The `verifiedAdvanceToNext` sync mechanism only fires in playlist mode at item transitions. Layout and scene modes have no transition points, so content re-sync for those modes relies on the 30-second polling cycle and `needs_refresh` flag rather than immediate transition-triggered sync.

**Impact:** Up to 30 seconds additional lag for layout/scene mode re-sync vs near-instant for playlist mode. Functional correctness is maintained.

#### INT-04 (Informational) — Extra RPC Per Heartbeat Cycle

`usePlayerHeartbeat` makes a separate `checkDeviceRefreshStatus` RPC call after `updateDeviceStatus` to check `needs_refresh`. The `update_device_status` SQL already reads this flag internally but does not return it. Optimization opportunity: return `needs_refresh` in the RPC response and eliminate the extra round-trip.
