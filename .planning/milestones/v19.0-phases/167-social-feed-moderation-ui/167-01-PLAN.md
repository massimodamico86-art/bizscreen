    2. Implement `fetchPendingModerationItems`:
       ```js
       export async function fetchPendingModerationItems() {
         const ownerId = await getEffectiveOwnerId();
         if (!ownerId) {
           console.warn('[socialFeedModerationService] No owner id; returning empty list.');
           return [];
         }

         // Left join social_feed_moderation via PostgREST embedding; keep only rows
         // where the moderation row is absent (i.e., moderation.id IS NULL → pending).
         const { data, error } = await supabase
           .from('social_feeds')
           .select(`
             id,
             provider,
             content_text,
             media_url,
             media_type,
             thumbnail_url,
             permalink,
             author_name,
             author_avatar,
             posted_at,
             created_at,
             moderation:social_feed_moderation!left(id)
           `)
           .eq('tenant_id', ownerId)
           .is('moderation.id', null)
           .order('posted_at', { ascending: false });

         if (error) {
           throw new Error(`Failed to fetch pending moderation items: ${error.message}`);
         }

         // Strip the embedded `moderation` key before returning — callers only need the feed shape.
         return (data || []).map(({ moderation, ...rest }) => rest);
       }
       ```

    5. Implement the shared internal helper `upsertModeration(feedId, approved)`:
       ```js
       async function upsertModeration(feedId, approved) {
         if (!feedId) {
           throw new Error('feedId is required');
         }
         const ownerId = await getEffectiveOwnerId();
         if (!ownerId) {
           throw new Error('No authenticated tenant context');
         }
         const { data: userResult } = await supabase.auth.getUser();
         const moderatedBy = userResult?.user?.id ?? null;

         const { data, error } = await supabase
           .from('social_feed_moderation')
           .upsert(
             {
               tenant_id: ownerId,
               feed_id: feedId,
               approved,
               moderated_by: moderatedBy,
             },
             { onConflict: 'tenant_id,feed_id' }
           )
           .select()
           .single();

         if (error) {
           const verb = approved ? 'approve' : 'reject';
           throw new Error(`Failed to ${verb} moderation item: ${error.message}`);
         }
         return data;
       }
       ```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-167-01 | Information Disclosure | fetchPendingModerationItems | mitigate | Query filters `social_feeds` by `tenant_id = ownerId` AND RLS policies on both tables require authenticated user (migration 081 lines 167-170, 185-188). Return early with [] when ownerId is null. |
| T-167-02 | Tampering | approveModerationItem / rejectModerationItem | mitigate | Service always sets `tenant_id` from `getEffectiveOwnerId()` (not from caller-supplied value), preventing cross-tenant writes. | T-167-03 | Repudiation | Moderation audit trail | accept | `moderated_by` and `moderated_at` columns are written on every upsert (migration 081 schema). No additional audit logging added in this phase — accept as sufficient for ASVS L1. |
| T-167-04 | Elevation of Privilege | Bypass of moderation state | mitigate | Pending detection uses `.is('moderation.id', null)` via PostgREST left-embed; cannot be subverted by client payload. All writes go through the `upsertModeration` helper which hard-codes approved as a boolean. |
| T-167-05 | Denial of Service | Unbounded queue growth | accept | `fetchPendingModerationItems` has no LIMIT; accept for now — if pending counts grow pathologically in production, Plan 02 or a follow-up phase can add pagination. Low severity for ASVS L1. |

No HIGH severity unmitigated threats remain.
