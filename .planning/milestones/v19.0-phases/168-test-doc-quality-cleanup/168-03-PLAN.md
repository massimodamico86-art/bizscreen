### Step 3 — Triage each failure as test-side drift OR product bug

For each failing test in `/tmp/168-03-run1.log`:

Classify per D-08 rubric:

| Signal | Classification |
|--------|----------------|
| Selector doesn't match any DOM node because the UI element was renamed/restructured (inspect trace with `npx playwright show-trace`) | **test-side drift** → fix forward |
| Helper function signature changed since Phase 162 (`helpers.js` API drift) | **test-side drift** → fix forward |
| Timing-sensitive `waitFor` too short for current dev-server boot | **test-side drift** → fix forward |
| Restored spec calls a helper that no longer exists at all | **test-side drift** → fix forward (add import or patch helper) |
| App surfaces an error state, returns wrong data, crashes, or violates documented behavior — and the test assertion is correct | **product bug** → defer |
| Widget renders incorrectly, playlist save persists wrong fields, auth flow broken in product code | **product bug** → defer |
| Feature is missing entirely (route 404, button not rendered in a supported state) | **product bug** → defer (or surface to user if ambiguous) |

Save the classification table to `/tmp/168-03-triage.md` with one row per failing test:


### Step 4 — Fix-forward loop (test-side drift only)

For EACH test-side drift failure:

1. Make the minimum targeted fix in the relevant test file (selector update, helper patch, increase a single `waitFor` timeout). Do NOT rewrite test logic. Do NOT change product code.
2. Re-run the affected spec:
   ```bash
   npx playwright test <spec-path> --reporter=list
   ```
3. If green, commit atomically:
   ```bash
   git add <file>
   git commit -m "fix(168-03): stabilize <test name> — <selector/helper/timing> drift since Phase 162"
   ```
4. Move to next failure.

Hard stop: if a fix attempt introduces a new failure, or if the same test fails for a different test-side reason twice in a row, escalate to user.
