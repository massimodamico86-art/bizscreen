# Phase 166.2: Code Review Report

**Reviewed:** 2026-04-12
**Depth:** standard
**Files Reviewed:** 1
**Status:** issues_found

## Summary

Migration 110 redefines `public.can_access_template(uuid)` and
`public.clone_template_to_scene(uuid, text)` using additive `CREATE OR REPLACE`,
removing the invalid `profiles.plan_tier` reference that caused 42703 errors.
All referenced identifiers (`template_library`, `template_library_slides`,
`template_enterprise_access`, `scenes`, `scene_slides`, `profiles.role`,
`profiles.id`) resolve against prior migrations. SECURITY DEFINER + SET
search_path = public is preserved on both functions, GRANT EXECUTE re-issued,
and D-01/D-02 decisions are implemented consistently.

No security regressions vs. migration 080. The two findings below concern (a)
internal inconsistency between the two functions' super_admin bypass placement
(behavioral-equivalent but subtle) and (b) a stale user-facing error message.
Info items note minor maintainability concerns.

No Critical findings.

## Warnings

### WR-01: Super-admin bypass ordering differs between the two functions

**File:** `supabase/migrations/110_fix_can_access_template.sql:57-64, 134-139`

**Issue:** `can_access_template` performs the `super_admin` check **before**
the license branching (returning `true` immediately). `clone_template_to_scene`
performs the `super_admin` check **after** the license branching, as a
fallback when `v_has_access` is still false. The end state is equivalent today
(both grant super_admin access to every license class, including enterprise
without an allow-list row), but the divergence is easy to break accidentally:

- In `clone_template_to_scene`, the `NOT v_has_access` guard means a future
  license-branch change that sets `v_has_access := true` for something else
  would short-circuit the super_admin check silently.
- Anyone reading the two functions side-by-side will reasonably assume a
  semantic difference exists where none is intended.

**Fix:** Move the `super_admin` check to the top of `clone_template_to_scene`
(immediately after the `template_library` lookup), mirroring
`can_access_template`. Example:

```sql
-- Super admins always have access (checked before license gating)
SELECT (role = 'super_admin') INTO v_has_access
FROM profiles
WHERE id = v_user_id;

IF NOT v_has_access THEN
  IF v_template.license IN ('free', 'pro') THEN
    v_has_access := true;
  ELSIF v_template.license = 'enterprise' THEN
    SELECT EXISTS (
      SELECT 1 FROM template_enterprise_access
      WHERE template_id = p_template_id
      AND tenant_id = v_user_id
    ) INTO v_has_access;
  END IF;
END IF;
```

This also avoids the subtle bug where the current `SELECT role = 'super_admin'
INTO v_has_access FROM profiles WHERE id = v_user_id` returns NULL (not false)
for a user with no `profiles` row, leaving `v_has_access` as NULL and
tripping the subsequent `IF NOT v_has_access` guard into raising — correct
behavior, but by accident.

### WR-02: Access-denied error message still references "plan tier"

**File:** `supabase/migrations/110_fix_can_access_template.sql:142`

**Issue:** The exception message reads
`'Access denied: insufficient plan tier for this template'`, but per D-01
plan-tier gating has been explicitly removed. The only remaining path to this
error is a non-allow-listed user attempting an `enterprise` template. The
message will confuse support and users ("I have a Pro plan, why does it say
insufficient tier?") and contradicts the migration's own D-01 comment block.

**Fix:** Update the message to reflect the actual gate:

```sql
RAISE EXCEPTION 'Access denied: this template requires an enterprise allow-list grant';
```

This is the only user-visible string in the migration, so getting it right
here avoids a follow-up migration later.

## Info

### IN-01: Cloned scene inherits template metadata but not industry-specific defaults

**File:** `supabase/migrations/110_fix_can_access_template.sql:146-159`

**Issue:** `clone_template_to_scene` copies `v_template.industry` into
`scenes.business_type` and `v_template.metadata` into `scenes.settings`, but
doesn't copy fields that may exist on newer `template_library` rows (e.g.
`layout_id`, default playlists). This matches migration 080 behavior and is
not a regression — flagging only so a future maintainer extending
`template_library` remembers this INSERT list is not auto-maintained.

**Fix:** None required for this phase. Document in `template_library` comments
that cloning is a hand-maintained projection.

### IN-02: `install_count` increment not idempotent on retry

**File:** `supabase/migrations/110_fix_can_access_template.sql:186-189`

**Issue:** The `UPDATE template_library SET install_count = install_count + 1`
runs inside the same transaction as the scene insert and slide copy. If the
client retries a failed call (e.g. timeout mid-function), PostgREST will
produce two inserts and two counter bumps on success. Not a correctness bug
(each clone really did create a scene), but the counter will drift above
"distinct users who installed." Pre-existing behavior from migration 080; no
worse after this migration.

**Fix:** None for 166.2. If analytics cares, track installs in a separate
`template_installs(template_id, user_id, installed_at)` table and derive
counts from `COUNT(DISTINCT user_id)`.

### IN-03: Comment references migration 017 for paid-entitlement re-gating

**File:** `supabase/migrations/110_fix_can_access_template.sql:69-70`

**Issue:** The D-01 re-gating hint says "tables exist in migration 017" but
doesn't specify which ones. Future maintainers will have to grep. Minor
doc-friction only.

**Fix:** Optionally expand to
`-- (subscriptions, plans — migration 017_plans_and_subscriptions.sql)` for
discoverability. Skip if phase budget is closed.
