# Phase 166.2: Fix `can_access_template` RPC - Context

**Gathered:** 2026-04-12
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase redefines two Postgres functions in a new migration so the
Template Marketplace RPC stops returning HTTP 400 and returns the 9 seeded
free-license rows to any authenticated client. Scope ends at **SQL-level
verification** of the fixed RPC; resuming Phase 166's four paused UAT
checkpoints is a manual follow-up driven by `/gsd-verify-work 166`.

</domain>

<decisions>
## Implementation Decisions

### Fix Strategy
- **D-01:** Remove plan-tier gating from the access check for `license = 'pro'`
  templates. After the fix, `pro` templates are treated identically to `free`:
  accessible to any authenticated user. Rationale: no `plan_tier` column exists,
  no subscription UI exists to populate one, and all 9 seeded rows are `free`
  today — the gate has been dead code since migration 080 shipped.
- **D-02:** Keep enterprise allow-list gating intact. `license = 'enterprise'`
  access continues to require an entry in `template_enterprise_access` for the
  caller's `tenant_id`, with the existing super_admin bypass preserved.

### Fix Scope
- **D-03:** Fix **both** offending functions in a single migration:
  - `can_access_template(uuid)` at
    [supabase/migrations/080_template_marketplace.sql:294-345](../../../supabase/migrations/080_template_marketplace.sql#L294-L345)
    (the README-named blocker).
  - `clone_template_to_scene(uuid, text)` at
    [supabase/migrations/080_template_marketplace.sql:175-284](../../../supabase/migrations/080_template_marketplace.sql#L175-L284),
    which references the same non-existent `profiles.plan_tier` column at line 208
    and would 42703 the moment Phase 166's "Apply & Create" UAT checkpoint fires.

### Migration Convention
- **D-04:** Ship as a new additive migration `110_fix_can_access_template.sql`
  using `CREATE OR REPLACE FUNCTION` for both functions. Do NOT edit
  `080_template_marketplace.sql` in place — edits to already-applied migrations
  do not re-run on remote Supabase environments. Follows the additive pattern
  established by Phase 166.1's seed migration (`109_seed_template_library_dev.sql`).

### Verification
- **D-05:** Gate phase completion on a SQL-level check:
  call `get_marketplace_templates()` as an authenticated (non-super-admin) role
  and assert:
  1. No error raised (i.e. no `42703: column "plan_tier" does not exist`).
  2. Exactly 9 rows returned (the seeded free-license set from migration 109).
  3. `can_access = true` on every returned row (free license path).

### Phase Boundary / Next Step
- **D-06:** Phase 166.2 ends at SQL verification. Resuming Phase 166's four
  paused UAT checkpoints (color preview, logo upload, Apply & Create, no-SVG
  fallback) is a manual operator step via `/gsd-verify-work 166` after this
  phase closes.

### Claude's Discretion
- Exact SQL body of the new `can_access_template` (likely collapses to:
  super_admin → true, free|pro → true, enterprise → allow-list lookup).
- Exact SQL body of the new `clone_template_to_scene` (retain structure;
  drop `v_user_plan` variable and its lookup; simplify the pro branch).
- Whether to drop the now-unused `v_user_plan` variable / plan_tier select
  or leave a COALESCE(NULL) stub for git-diff minimalism.
- Whether to re-`GRANT EXECUTE` after `CREATE OR REPLACE` (safe either way).
- Plan-level concerns (file naming conventions, commit message, whether to
  run `supabase db push` in execute-phase).

</decisions>

## Existing Code Insights

### Reusable Assets
- `CREATE OR REPLACE FUNCTION` is the established pattern for function changes
  in this repo (used throughout `supabase/migrations/`).
- `template_enterprise_access` table + its existing RLS stay untouched — the
  enterprise allow-list lookup already works as-is.
- `profiles.role = 'super_admin'` bypass is already present in both functions
  and must be preserved.

### Established Patterns
- Additive migration numbering: the next free slot is `110` (Phase 166.1 used 109).
- Function migrations use `SECURITY DEFINER` + `SET search_path = public`;
  migration 110 must keep these attributes.
- `GRANT EXECUTE ON FUNCTION ... TO authenticated` is re-issued after every
  definition in 080 — safe to re-issue in 110.

### Integration Points
- `get_marketplace_templates()` (defined in 080, unchanged by this phase)
  calls `can_access_template(tl.id)` per-row at
  [supabase/migrations/080_template_marketplace.sql:399](../../../supabase/migrations/080_template_marketplace.sql#L399).
  That call site is the one that currently raises 42703.
- Client callers: [src/pages/TemplateMarketplacePage.jsx](../../../src/pages/TemplateMarketplacePage.jsx)
  and [src/services/marketplaceService.js](../../../src/services/marketplaceService.js)
  invoke the RPC via PostgREST; no client changes needed — they already handle
  the happy path.

<specifics>
## Specific Ideas

- Use file name `110_fix_can_access_template.sql` (matches README hint).
- Migration should contain exactly two `CREATE OR REPLACE FUNCTION` blocks
  (plus their GRANTs) and nothing else — no schema changes, no data changes.
- For verification SQL, prefer a single session that `SET LOCAL role authenticated;`
  (or uses a test user JWT) and runs `SELECT * FROM get_marketplace_templates();`
  expecting `count(*) = 9` with no error.

</specifics>

<deferred>
## Deferred Ideas

- **Restore plan-tier gating for `pro` templates.** If a paid-tier product gate
  becomes a real requirement, reintroduce it by joining
  `subscriptions.plan_id → plans.slug` inside `can_access_template` (the tables
  already exist in migration 017). Deferred because: (a) no subscription UI
  exists, (b) no `pro` templates are seeded, (c) the current `plan_tier` column
  never existed. Route this to the backlog (`/gsd-add-backlog`) for the next
  time billing/entitlements work lands on the roadmap.
- **Second-pass cleanup of `clone_template_to_scene`.** After D-03 lands, the
  function's `v_has_access` branch logic simplifies to the point where the
  whole "Check license access" block could be inlined or replaced with a call
  to `can_access_template(p_template_id)`. Out of scope here to keep the diff
  surgical.

</deferred>
