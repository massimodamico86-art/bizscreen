## Task 1: Create migration file 109_seed_template_library_dev.sql

**Type:** auto
**Status:** complete
**Commit:** a038ffc7

### What changed

One new file: `supabase/migrations/109_seed_template_library_dev.sql` (194 insertions).

Content shape:
- First `DO $$` block iterates an 8-element `jsonb_build_array` of seed specs
  (retail, restaurant, corporate, healthcare, education, hospitality, fitness,
  events), for each doing an idempotency-guarded `INSERT INTO template_library`
  followed by an idempotency-guarded `INSERT INTO template_library_slides` at
  `position = 0`. Every populated SVG is a hand-authored string with:
  - One `<rect>` background fill
  - At least two `<text>` elements with distinct `fill=` colors
  - One `<image data-logo-placeholder="true" ...>` for logo swap
- Second `DO $$` block inserts the 9th fallback-test row
  (`name = 'Empty Template (Fallback Test)'`) whose `metadata` intentionally
  OMITS `svgContent`, and its slide's `design_json` likewise omits it.
- Third `DO $$` block prints a `RAISE NOTICE` with row counts for immediate
  post-apply verification.

All 9 rows are tagged `metadata->>'seed_source' = 'dev-166.1'` which is the
predicate the idempotency guards and cleanup queries use.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] `supabase db push` replaced with `docker exec psql`**

- **Found during:** Task 2 Step 1.
- **Issue:** The plan's `<action>` block instructed the executor to run
  `supabase db push --yes`. This command targets the REMOTE linked Supabase
  project and requires `supabase login` or `SUPABASE_ACCESS_TOKEN`. For a local
  dev stack (which this worktree runs — `VITE_SUPABASE_URL` points at
  127.0.0.1:54321), `db push` is the wrong channel and would attempt to write
  the seed to prod/staging rather than the local Postgres.
- **Fix:** Followed the orchestrator's environment-notes fallback ladder:
  1. Tried `npx supabase migration up --local` — failed because the CLI's
     internal `schema_migrations` table references 50+ migrations not present
     in this divergent worktree.
  2. Fell back to `docker exec -i supabase_db_bizscreen psql -U postgres -d postgres < supabase/migrations/109_seed_template_library_dev.sql`.
  The migration file itself is unchanged; only the application channel differs.
- **Files modified:** None (only the DB state changed).
- **Commit:** n/a (DB-only change, no source impact).
