<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user click → URL writer | User clicks button; the sceneId + slideId values come from React state (already trusted within the component) |
| URL → TemplateGalleryPage | Plan 06's gallery reads ?editorReturn=1 & returnSceneId & slideId — these are user-controllable via direct URL editing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-174-05-01 | Tampering | URL params (returnSceneId, slideId) editable by user | accept | The DB RPC apply_template_to_active_slide enforces auth.uid() ownership of both scene and slide. A malicious user setting `?returnSceneId=<other-tenant-scene>` triggers a `Scene not found or access denied` RAISE EXCEPTION on the round-trip Apply (Plan 02 mitigation T-174-02-01). Defence-in-depth at the DB layer makes client-side URL trust unnecessary. |
| T-174-05-02 | Information Disclosure | sceneId/slideId in URL bar | accept | UUIDs are already in the URL when the user is editing a scene (the page key `scene-editor-<sceneId>`). Showing them in query params is no additional disclosure. |
| T-174-05-03 | Denial of Service | rapid clicks of Browse Templates trigger many navigate() calls | accept | navigate() with `{ replace: true }` does not stack history entries; rapid clicks just re-render the destination once. No DoS surface. |

ASVS Level 1: V4 (Access Control) — DB-side ownership check is the real trust boundary; this plan adds none of its own checks (correctly).
</threat_model>
