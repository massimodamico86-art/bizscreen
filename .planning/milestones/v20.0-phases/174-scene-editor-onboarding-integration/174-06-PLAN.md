## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-174-06-01 | Tampering | URL params (editorReturn, returnSceneId, slideId) editable by user | accept | DB RPC enforces ownership of returnSceneId AND that returnSlideId belongs to that scene (Plan 02 mitigations T-174-02-01). User editing URL to a foreign scene id triggers `Scene not found or access denied`. |
| T-174-06-02 | Tampering | Polotno bypass — user removes editorReturn param to force gallery to show polotno | mitigate | Removing editorReturn = standard gallery mode (no Use Template button → uses 'Apply to new scene' which is the safe Phase 172 path, creating a NEW scene). The only way to invoke editor-return apply is to satisfy ALL three URL params; even then, the RPC's `editor_type != 'svg'` rejection (Plan 02 mitigation T-174-02-02) blocks polotno. |
| T-174-06-03 | XSS | data-tour attribute values | accept | All three values are static literals (`"filter-bar"`, `"search-input"`, `"first-card"`). No user input flows into them. |
| T-174-06-04 | Information Disclosure | StarterPacksStrip hidden in editorReturn might leak that the user is mid-edit | accept | Visibility of UI elements based on URL params is not a security signal worth protecting (no PII; no sensitive ownership data). |

ASVS Level 1: V4 (Access Control) — relies on Plan 02 RPC mitigations; this plan adds correct UI-layer routing without weakening DB checks.
