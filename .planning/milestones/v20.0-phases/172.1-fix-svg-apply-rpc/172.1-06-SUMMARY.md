## Three fixes applied to get here

**Fix 1 — Migrations applied to local DB (127.0.0.1:54321).**
`.env.local` overrides `VITE_SUPABASE_URL` to the local Docker Supabase container, not the linked remote project `gdxizdiltfqeugbsgtpx`. Plan 03's production push only touched the remote. Without also applying to local, the dev server's app had `svg_content = NULL` on all rows and no `clone_svg_template_to_scene` function. Fix: `docker exec -i supabase_db_bizscreen psql ... < supabase/migrations/169_...sql` + same for 170. After this step, 2 → 3 tests passing.

**Fix 2 — Migration 169 dollar-quote delimiter newlines (commit [c71f994b](../../commits/c71f994b)).**
Original 169 was generated with `echo 'SET svg_content = $svg$'` + `cat file` + `echo '$svg$'`, which produced `SET svg_content = $svg$\n<?xml...</svg>\n$svg$` — storing content with a leading `\n`. The preview pipeline's `DOMParser` on `'image/svg+xml'` wouldn't find `<svg>` when fed a leading-newline payload, raising `'Invalid SVG content'`. Fix: wrap every UPDATE's RHS in `TRIM(BOTH E' \t\r\n' FROM $svg$...$svg$)`. Also applied as a one-time cleanup to the live prod DB (remote + local both got TRIMmed rows).

**Fix 3 — Modal inner height constraint (commit [430934b1](../../commits/430934b1)).**
The real UI culprit for 4 of the 7 failing tests. The Modal panel has `max-h-[90vh]` but no explicit height; its inner flex-column child used `h-full` which resolved against the parent's computed (content-based) height instead of the 90vh cap. Result: at 1280×720 viewport, the modal inner measured 828px tall — pushing the sticky Apply button to y=796, 76px below the visible viewport. Fix: replace `h-full` with `h-[calc(100vh-2rem)] sm:h-[90vh]` on the inner div + added `min-h-0` to the right panel. Apply button now at y=616 (in viewport).
