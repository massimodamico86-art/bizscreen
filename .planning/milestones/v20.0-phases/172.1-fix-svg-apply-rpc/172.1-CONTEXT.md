# Phase 172.1: Fix SVG Apply RPC - Context

**Gathered:** 2026-04-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Resolve the architectural mismatch between the `gallery_templates` VIEW (UNION of `svg_templates` + `template_library`) and the new `clone_template_with_customization` RPC shipped in migration 168 — which reads only from `template_library` and therefore fails with "Template not found or inactive" for every SVG template. Surfaced by live E2E UAT of Phase 172 on 2026-04-21: all 24 active SVG templates in `svg_templates` fail Apply.

**In scope:**
- A new sibling RPC that handles the `editor_type='svg'` Apply path atomically.
- A data backfill so `svg_templates.svg_content` is populated for every active row.
- The one-line client dispatch swap in `templateApplyService.js` so the `svg` branch calls the new RPC.
- Re-running Phase 172's existing 7-test Playwright `preview-apply.spec.js` against live credentials to confirm green.

**Not in this phase:**
- Any change to migration 168 (`clone_template_with_customization`) — treated as immutable.
- Any change to the Polotno Apply path (`clone_template_to_scene`) — explicitly out of scope.
- TPRV-F1 Polotno in-preview QuickCustomize (deferred to v20.1).
- Adding a `license` column to `svg_templates` (future work if per-template licensing becomes a product need).
- Client-side runtime fetch of `svg_url` (backfill covers all current rows; revisit if a URL-only admin-upload path lands).

</domain>

<decisions>
## Implementation Decisions

### RPC Architecture

- **D-01:** Ship a **new sibling RPC `clone_svg_template_to_scene`** in a new migration. Client dispatches by `editor_type` in `src/services/templateApplyService.js` — the `svg` branch calls the new RPC, the `polotno` branch continues to call `clone_template_to_scene`, and migration 168's `clone_template_with_customization` stays untouched. This keeps blast radius minimal and preserves symmetry with the existing `clone_template_to_scene` / `clone_template_with_customization` split.

- **D-02:** The SVG path is **atomic on the same TPRV-05 contract** as migration 168. The new function wraps its `INSERT INTO scenes` + single `INSERT INTO scene_slides` in one PL/pgSQL function (PL/pgSQL runs in the caller's transaction, so this is atomic by construction). No client-side UPDATE after the insert.

- **D-03:** **Migration 168 is immutable.** It has been applied to the production project (`gdxizdiltfqeugbsgtpx`) and passed atomicity audit. 172.1 adds new migration files; it does not `CREATE OR REPLACE` migration 168's function body.

- **D-04:** **RPC signature mirrors migration 168's 3-arg shape** for dispatch symmetry:
  ```
  clone_svg_template_to_scene(
    p_template_id     uuid,
    p_scene_name      text DEFAULT NULL,
    p_customized_svg  text DEFAULT NULL
  ) RETURNS uuid
  ```
  The only call-site difference between the two RPCs becomes the function name.

### SVG Content Source

- **D-05:** **SVG content resolution priority (inside the new RPC):**
  1. `p_customized_svg` if non-null → patched into `design_json.svgContent` via `jsonb_set` with `to_jsonb(...)` (same pattern as migration 168:132-138).
  2. Otherwise → `svg_templates.svg_content` column value.
  3. Otherwise → `RAISE EXCEPTION 'Template has no SVG body'` (deterministic, surfaces missing seed data immediately).

  No server-side HTTP fetch of `svg_url`. The RPC remains a dumb persistor per Phase 172 D-10.

- **D-06:** **Backfill `svg_content` for all 24 active `svg_templates` rows** via a data migration. Read the 12 unique static SVG files under `public/templates/svg/**/design.svg` (+ `menu-design.svg`) into the `svg_content` column. UPDATE matches rows by `svg_url` (all rows reference one of those 12 paths), so the same UPDATE covers both the 097-seeded (no-slug) and 167-seeded (with-slug) row sets. **Idempotent:** UPDATE uses `WHERE svg_content IS NULL` guard. No DOWN migration (matches project convention).

- **D-07:** **Verified row counts:** `097_seed_svg_templates.sql` inserts 12 rows (no `slug` — pre-dates migration 167). `167_gallery_templates_view_and_rls.sql` inserts 12 rows with `slug` via `ON CONFLICT (slug) WHERE slug IS NOT NULL DO NOTHING` — since the 097 rows have `slug IS NULL` they do not conflict. Net: **24 rows in `svg_templates`**, matching the ROADMAP's "24 active SVG templates" figure. All 24 point to one of 12 unique static files under `public/templates/svg/`.

- **D-08:** **QuickCustomizePanel flow is unchanged** in this phase. Once `svg_content` is backfilled, the panel's existing `parseSvgForCustomize(template.svg_content)` call path works immediately. No new client-side fetch of `svg_url` is added — that's deferred until a URL-only template admin path lands.

### License / Access Gate

- **D-09:** **No license gate** on the SVG Apply path. `svg_templates` has no `license` column; the base-table SELECT RLS is already `is_active = TRUE AND (tenant_id IS NULL OR created_by = auth.uid())` (migration 167:39-45, which closed TDAT-03). The new RPC mirrors that predicate exactly so it cannot grant more access than a plain SELECT on the same row.

- **D-10:** **RPC preamble is:**
  ```
  auth.uid() IS NOT NULL        → else RAISE 'Not authenticated'
  svg_templates row is_active=TRUE AND (tenant_id IS NULL OR created_by = auth.uid())
                                → else RAISE 'Template not found or inactive'
  super_admin bypass            → retained for parity with migration 168 (profiles.role = 'super_admin')
  ```
  No `template_enterprise_access` lookup (that table is keyed to `template_library.id`, not `svg_templates.id`).

- **D-11:** **Scope: any authenticated user for global + own SVG templates.** The RPC allows both `tenant_id IS NULL` (global seeded) and `created_by = auth.uid()` (user-owned, future path) rows. Cross-tenant reads remain blocked by the predicate. This matches the existing svg_templates SELECT RLS one-to-one.

### Client Dispatcher

- **D-12:** `src/services/templateApplyService.js` **`applyTemplate`** changes only the RPC name in the `svg` branch:
  - Before: `supabase.rpc('clone_template_with_customization', { p_template_id, p_scene_name, p_customized_svg })`
  - After:  `supabase.rpc('clone_svg_template_to_scene', { p_template_id, p_scene_name, p_customized_svg })`

  DOMPurify sanitization (D-17 from Phase 172) stays. The 500KB size cap (`SVG_MAX_BYTES`) stays. `editorRouteFor` stays (still returns `svg-editor?sceneId=${id}` — D-15 contract unchanged).

- **D-13:** `clone_template_with_customization` becomes **Polotno-only** in practice after this phase. Its `p_customized_svg` parameter stays in the signature (no DB change) but is never invoked with a non-null value by the client going forward. Documenting this — not removing it — since migration 168 is immutable (D-03) and a future Polotno-customize path could still use it.

### Migration Ordering

- **D-14:** **Two new migration files, in order** (planner confirms next available numbers at plan time):
  1. Data backfill of `svg_templates.svg_content` from static files.
  2. `CREATE OR REPLACE FUNCTION public.clone_svg_template_to_scene(...)` + `GRANT EXECUTE ... TO authenticated` + `COMMENT ON FUNCTION ...`.

  Backfill runs before the new RPC is shipped so the RPC never observes a row where both `p_customized_svg` and `svg_content` are null. Both are additive; no schema changes.

### Claude's Discretion

- **Scene field mapping** from an `svg_templates` row into the new `scenes` row — planner picks sensible defaults. Recommended:
  - `scenes.name` ← `COALESCE(p_scene_name, v_template.name || ' (Copy)')` (mirror 168:104)
  - `scenes.business_type` ← `v_template.category` (closest analog to `template_library.industry`)
  - `scenes.settings` ← `jsonb_build_object('width', v_template.width, 'height', v_template.height, 'orientation', v_template.orientation)`
  - `scenes.is_active` ← `true`
  - `scenes.tenant_id` ← `auth.uid()` (mirror 168:103)

- **`scene_slides` row shape** — single row, recommended:
  - `position` ← `0`
  - `title` ← `v_template.name`
  - `kind` ← `'default'` (align with `template_library_slides` default)
  - `design_json` ← `jsonb_build_object('svgContent', <resolved svg>)`
  - `duration_seconds` ← `10` (align with `template_library_slides.duration_seconds DEFAULT 10`)

- **Backfill strategy** — inline SQL `UPDATE` with `$$...$$`-quoted SVG bodies, or a Node script that generates the UPDATE statements from the files. Prefer whichever keeps the migration reviewable; 12 unique files × ~2–10KB each is small enough to inline if formatted readably.

- **Integration test shape** — add a vitest integration test paralleling `tests/integration/preview-apply/rpc-atomicity.test.js` but exercising `clone_svg_template_to_scene` (single RPC call, no follow-up UPDATE, error propagation on missing SVG body). Re-use the 7 existing Playwright `preview-apply.spec.js` tests against live credentials to prove the regression is closed.

- **Migration file names** — next available (likely `169_` and `170_`). Planner confirms at plan time and picks descriptive slugs.

<code_context>
## Existing Code Insights

### Reusable Assets
- Migration 168 is a near-complete blueprint — new RPC mirrors its preamble (auth check, NOT NULL check, FOUND / IS NULL guard), its `INSERT INTO scenes` shape, its `jsonb_set` patch pattern, and its `GRANT EXECUTE ... TO authenticated` footer. Only the source-table, license-gate, and slide-loop diverge.
- The existing client dispatcher in `templateApplyService.js` already branches on `template.editor_type`. The fix is a one-line RPC name swap in the `svg` branch.
- `svgCustomizeService` (Phase 172 Plan 01) keeps working once `svg_content` is backfilled — no re-architecture of the customize pipeline needed.
- `SvgEditorPage.jsx` `?sceneId=` branch (Phase 172 D-15) already reads `slide.design_json.svgContent` — the new RPC writes exactly that shape, so the editor load path is unchanged.

### Established Patterns
- RPCs return `uuid` (scene id); named-parameter calls from the client (`supabase.rpc(name, { p_... })`); `SECURITY DEFINER` + `SET search_path = public`; `GRANT EXECUTE ... TO authenticated` per function.
- Migrations are idempotent (`CREATE OR REPLACE FUNCTION`, UPDATE with `WHERE ... IS NULL` guards), no DOWN migrations.
- Client services are named-export JS modules; errors propagate; sanitization sits at the client boundary.

### Integration Points
- `svg_templates` table — both RPC reader and backfill target.
- `scenes` + `scene_slides` tables — write destinations (same as migration 168 Polotno path).
- `templateApplyService.applyTemplate` svg branch — single call-site that needs the new RPC name.
- `public/templates/svg/**` static files — read-only source for the backfill.

### Creative Options Enabled
- Because the VIEW already exposes `editor_type` and the client already dispatches on it, a new RPC slots in with near-zero architectural change. The "one RPC per source table" split is symmetric with `clone_template_to_scene` vs `clone_template_with_customization` and leaves room for future template sources (e.g., a hypothetical `figma_templates` table) without forcing a polymorphic dispatcher in SQL.
- Because backfill populates `svg_content` for every active row, QuickCustomizePanel's existing empty-state branch becomes unreachable for seeded data — future work that adds URL-only templates can still trigger it, keeping the fallback alive for forward-compat.

## Deferred Ideas

- **`license` column on `svg_templates`.** Adding per-template licensing to SVG would be a schema expansion with migration + backfill + admin UI implications. Not needed for this bugfix (D-09). Future phase if the product adds tiered SVG content.
- **Client-side runtime fetch of `svg_url`.** Only valuable when admin paths start inserting URL-only rows (no `svg_content`). Current seed data is fully backfilled, so this is forward-compat work. Revisit when a Phase 175-style content pipeline ships new URL-only templates.
- **Removing `p_customized_svg` from migration 168's signature.** Per D-03 migration 168 is immutable, and per D-13 the Polotno client will never invoke it with a non-null value going forward. Not a bug — just dead surface on a published function. Revisit only if a future Polotno customize flow explicitly needs the parameter to stay.
- **Refactoring `templateApplyService` to consolidate RPC names into a map.** Currently two `if (editor_type === ...)` branches. A lookup table would be cleaner but is premature with only two editor types. Revisit when a third is added.
