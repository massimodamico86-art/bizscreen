# Plan 172.1-01 Summary — Migration 169: Backfill svg_templates.svg_content

## What was built

`supabase/migrations/169_backfill_svg_content.sql` — a 77.2 KB data migration that inlines the verbatim contents of all 12 unique static SVG files under `public/templates/svg/**` into 12 URL-keyed, idempotent UPDATE blocks against `svg_templates.svg_content`. Each UPDATE hits both the 097-seeded (slug IS NULL) and 167-seeded (slug bearing) row for its `svg_url` — net 24 rows touched by 12 UPDATEs.

The migration closes with a self-asserting `DO $$ ... ASSERT v_missing = 0 ... END $$` block that raises and aborts the transaction if any active row still has `svg_content IS NULL` after the backfill. This is the D-14 ordering guard that prevents migration 170's RPC from shipping against an under-backfilled DB.

## How it works

- ASCII-bordered header cites D-06 (idempotency guard), D-07 (scope), and D-14 (ordering vs migration 170).
- 12 UPDATE blocks, each using `$svg$...$svg$` dollar-quoted strings so no inner SVG attribute single-quotes need escaping.
- Each UPDATE matches on `svg_url` (not `slug`), so it covers both row sets seeded by 097 and 167.
- Idempotency guard: `WHERE svg_content IS NULL` on every UPDATE — re-running produces zero new UPDATEs.
- Self-verification footer: `DO $$` block counts active rows with `svg_content IS NULL`, raises via `ASSERT v_missing = 0` if any remain.
- No DELETE, no INSERT, no ON CONFLICT, no DOWN migration — pure additive data fix.

## Notes for downstream plans

- **Plan 02 (migration 170):** `svg_templates.svg_content` is guaranteed populated for every active row once migration 169 applies (Plan 03 gate). Plan 02's RPC D-05 fallback #3 (`RAISE EXCEPTION 'Template has no SVG body'`) is therefore unreachable on seeded data.
- **Plan 03 (db push):** migration 169 must apply BEFORE 170. Because 169 lives at a lower number, `supabase db push --linked` will process it first in a single transaction. If 169's ASSERT fails, 170 never runs — exactly the D-14 contract.
- **Inline SQL size:** 77 KB file is reviewable in-editor. 12 verbatim SVG bodies total ~73 KB (largest: restaurant-menu/menu-design.svg at 28 KB). No minification was applied per the plan's explicit instruction — any byte drift would desync the backfill from the static assets served at build time.
