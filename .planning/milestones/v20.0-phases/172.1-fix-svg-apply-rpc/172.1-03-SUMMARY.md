# Plan 172.1-03 Summary — Live deploy of migrations 169 + 170

## What was done

Applied migrations 169 (svg_content backfill) and 170 (`clone_svg_template_to_scene` RPC) to the linked Supabase project `gdxizdiltfqeugbsgtpx` (BizScreen production).

## Push method

The Supabase CLI was not available on this shell's PATH (`which supabase` returned "not found") and `SUPABASE_ACCESS_TOKEN` was not exported. User authorized the alternative MCP path. Deploy was performed via the Anthropic `mcp__supabase__*` tools:

- **Migration 169 backfill** — sent as 12 separate `execute_sql` calls (one UPDATE per file, ordered smallest to largest), followed by one final `execute_sql` call for the self-asserting DO $$ footer. The ASSERT raised no error, confirming all 24 active rows have `svg_content IS NOT NULL`.
- **Migration 170 RPC** — sent as a single `apply_migration` call, which registered it in `supabase_migrations.schema_migrations` with version `20260422191755 clone_svg_template_to_scene`.

All SQL executed against the live production project.

### Why 169 was not sent via apply_migration

The 77 KB migration 169 exceeded the Read tool's 25 K-token limit for single-pass ingestion, so it was split into per-UPDATE chunks (each < 29 KB) and replayed individually via `execute_sql`. This means **169 is NOT registered in `schema_migrations`**. The data landed correctly, but a future `supabase db push --linked` from the developer's local CLI will see 169 as unapplied and try to reapply it. This is safe — 169 is idempotent (`WHERE svg_content IS NULL` on every UPDATE) and the ASSERT passes on already-satisfied state.
