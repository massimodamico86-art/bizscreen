### 1. `supabase/migrations/NNN_backfill_svg_content.sql` (NEW migration, data backfill)

**Primary analog for row shape:** `supabase/migrations/097_seed_svg_templates.sql`
**Primary analog for idempotency + embedded verification:** `supabase/migrations/167_gallery_templates_view_and_rls.sql`

**Why two analogs:** 097 establishes how seeded SVG rows reference `svg_url` paths; 167 establishes the idempotency guard (`ON CONFLICT ... DO NOTHING`, partial indexes) and the self-verifying `DO $$ ... ASSERT ...` block pattern. Backfill combines the URL set from 097 with an idempotency pattern more like 167.

**Header preamble pattern** (167:1-12 — use this shape, NOT 097's terse 4-line header):
```sql
-- ============================================================================
-- Migration NNN: Backfill svg_templates.svg_content from static files (Phase 172.1)
--
-- Addresses UAT regression surfaced in Phase 172 live E2E: all 24 active
-- svg_templates rows have svg_content = NULL, so the new SVG Apply RPC
-- (migration MMM) has no server-side fallback when p_customized_svg is null.
--
-- Scope (per 172.1-CONTEXT.md D-06/D-07):
--   * 24 rows in svg_templates (12 seeded by 097 with slug=NULL, 12 seeded
--     by 167 with slug bearing). Both sets reference the same 12 unique
--     static files under public/templates/svg/**.
--   * UPDATE matches by svg_url (not slug) so it covers both row sets.
--   * Idempotent: WHERE svg_content IS NULL guard (D-06).
--
-- Runs BEFORE migration MMM (the new RPC) so the RPC never observes a row
-- where both p_customized_svg and svg_content are null (D-14).
--
-- No DOWN migration (matches 080/110/167/168 convention).
-- ============================================================================
```

**UPDATE pattern — reference for idempotent backfill by `svg_url`:**
```sql
-- Pattern (12 blocks, one per static file):
UPDATE svg_templates
SET svg_content = $svg$<svg ...full-file-contents...></svg>$svg$
WHERE svg_url = '/templates/svg/restaurant-menu/menu-design.svg'
  AND svg_content IS NULL;
```

Notes:
- Use `$svg$...$svg$` dollar-quoted strings (PL/pgSQL convention in the codebase; avoids the need to escape inner single-quotes inside SVG attribute values).
- URL keys are the 12 unique paths listed in CONTEXT.md §Static SVG Assets.
- The UPDATE hits both the 097-seeded (slug IS NULL) and 167-seeded (slug bearing) rows because it keys off `svg_url`, not `slug` (D-06).

**Self-verifying `DO $$` footer pattern** — copy shape from 167:287-320:
```sql
-- ============================================================================
-- SELF-ASSERTING VERIFICATION — all active rows have svg_content populated
-- ============================================================================
DO $$
DECLARE
  v_missing INTEGER;
BEGIN
  SELECT COUNT(*) INTO v_missing
    FROM svg_templates
    WHERE is_active = TRUE
      AND svg_content IS NULL;
  ASSERT v_missing = 0,
    format('expected 0 active svg_templates rows with svg_content IS NULL after backfill, got %s', v_missing);
END $$;
```

**What NOT to copy from 097** (097 is pre-167; its pattern is out-of-date in two ways):
- 097 does a `DELETE FROM svg_templates WHERE tenant_id IS NULL` at line 7 before inserting — **do NOT delete; backfill must be additive**.
- 097's `ON CONFLICT DO NOTHING` at line 191 is a no-op fallback since that migration uses a bare INSERT without conflict target. Prefer 167's partial-index conflict target pattern if any INSERT is needed (none is needed here — this migration is UPDATE-only).

### 2. `supabase/migrations/MMM_clone_svg_template_to_scene.sql` (NEW migration, atomic RPC)

**Analog:** `supabase/migrations/168_clone_template_with_customization.sql` (near-complete blueprint per CONTEXT.md §Reusable Assets)

**Header preamble pattern** (168:1-38 — copy shape; substitute svg_templates story):
```sql
-- ============================================================================
-- Migration MMM: Atomic clone-svg-template-to-scene RPC (Phase 172.1)
--
-- Sibling RPC to migration 168's clone_template_with_customization, but
-- sources rows from svg_templates (not template_library). Fixes the
-- "Template not found or inactive" regression surfaced by Phase 172 live UAT
-- (2026-04-21): migration 168's function reads only from template_library,
-- so every editor_type='svg' Apply failed (all 24 active svg_templates rows).
--
-- Signature mirrors migration 168 for dispatch symmetry (172.1 D-04):
--   clone_svg_template_to_scene(uuid, text, text) RETURNS uuid
--
-- Operation is a single PL/pgSQL function. PL/pgSQL runs in the caller's
-- transaction, so scene INSERT + scene_slides INSERT are atomic by
-- construction. Mirrors migration 168's TPRV-05 atomicity contract.
--
-- License gate: NONE (172.1 D-09). svg_templates has no `license` column.
-- Access is restricted to the same predicate as svg_templates SELECT RLS
-- (migration 167:39-45, which closed TDAT-03): is_active=TRUE AND
-- (tenant_id IS NULL OR created_by = auth.uid()). super_admin bypass
-- retained for parity with migration 168.
--
-- SVG content resolution priority (172.1 D-05):
--   1. p_customized_svg if non-null → jsonb_set design_json.svgContent
--   2. svg_templates.svg_content column value
--   3. RAISE EXCEPTION 'Template has no SVG body' (deterministic)
--
-- Migration 168 is immutable (172.1 D-03) — this adds a sibling, never
-- CREATE OR REPLACEs 168's body.
--
-- No DOWN migration (matches 080/110/167/168 convention).
-- ============================================================================
```

**Function signature pattern** (168:40-49 — copy verbatim, rename):
```sql
CREATE OR REPLACE FUNCTION public.clone_svg_template_to_scene(
  p_template_id uuid,
  p_scene_name text DEFAULT NULL,
  p_customized_svg text DEFAULT NULL
)
RETURNS uuid
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
DECLARE
  v_user_id uuid;
  v_template svg_templates%ROWTYPE;      -- CHANGED from template_library
  v_has_access boolean := false;
  v_new_scene_id uuid;
  v_resolved_svg text;                    -- NEW local for D-05 resolution
BEGIN
```

**Auth preamble pattern** (168:56-61 — copy verbatim):
```sql
  -- Get authenticated user
  v_user_id := auth.uid();
  IF v_user_id IS NULL THEN
    RAISE EXCEPTION 'Not authenticated';
  END IF;
```

**Template-fetch pattern — DIVERGES from 168** (168:63-70 selects from template_library with `is_active = true` only; new RPC must apply the full svg_templates RLS predicate per D-10/D-11):
```sql
  -- Get template — mirrors svg_templates SELECT RLS (migration 167:39-45)
  SELECT * INTO v_template
  FROM svg_templates
  WHERE id = p_template_id
    AND is_active = TRUE
    AND (tenant_id IS NULL OR created_by = v_user_id);

  IF v_template IS NULL THEN
    RAISE EXCEPTION 'Template not found or inactive';
  END IF;
```

**super_admin bypass pattern** (168:84-89 — retained per D-10 for parity; applied AFTER the RLS-predicate fetch):
```sql
  -- Super admin bypass: if the RLS-scoped SELECT above returned nothing,
  -- allow super_admins to fetch the row unrestricted (parity with migration 168).
  IF v_template IS NULL THEN
    SELECT role = 'super_admin' INTO v_has_access
    FROM profiles
    WHERE id = v_user_id;

    IF v_has_access THEN
      SELECT * INTO v_template
      FROM svg_templates
      WHERE id = p_template_id AND is_active = TRUE;
    END IF;

    IF v_template IS NULL THEN
      RAISE EXCEPTION 'Template not found or inactive';
    END IF;
  END IF;
```
*Planner note:* The IF-NULL-THEN-super_admin-retry is the cleanest mapping of 168's "super_admin bypasses license gate" onto "super_admin bypasses tenant predicate." CONTEXT.md D-10 says retain the bypass "for parity with migration 168." Either this two-step form or a compound WHERE in the primary fetch is acceptable; this form preserves 168's readable "RLS check, then bypass" narrative.

**SVG content resolution pattern** (D-05 — no direct 168 analog; 168 only patches, it never falls back):
```sql
  -- Resolve SVG body per 172.1 D-05:
  --   1. p_customized_svg if non-null  → client-sanitized customized SVG
  --   2. svg_templates.svg_content     → backfilled static-file content
  --   3. RAISE EXCEPTION               → deterministic surface of missing seed
  IF p_customized_svg IS NOT NULL THEN
    v_resolved_svg := p_customized_svg;
  ELSIF v_template.svg_content IS NOT NULL THEN
    v_resolved_svg := v_template.svg_content;
  ELSE
    RAISE EXCEPTION 'Template has no SVG body';
  END IF;
```

**Scene INSERT pattern** (168:95-109 — copy shape; field mapping per CONTEXT.md §Claude's Discretion):
```sql
  -- Create new scene (field mapping per 172.1-CONTEXT.md §Claude's Discretion)
  INSERT INTO scenes (
    tenant_id,
    name,
    business_type,
    settings,
    is_active
  ) VALUES (
    v_user_id,                                                    -- mirror 168:103
    COALESCE(p_scene_name, v_template.name || ' (Copy)'),         -- mirror 168:104
    v_template.category,                                          -- closest analog to template_library.industry
    jsonb_build_object(
      'width',       v_template.width,
      'height',      v_template.height,
      'orientation', v_template.orientation
    ),
    true                                                          -- mirror 168:107
  )
  RETURNING id INTO v_new_scene_id;
```

**Slide INSERT pattern — DIVERGES from 168** (168:114-143 loops over `template_library_slides`; new RPC inserts a SINGLE slide row — svg_templates has no companion slides table):
```sql
  -- Single slide row (svg_templates has no companion slides table — unlike
  -- template_library → template_library_slides). Shape per 172.1 §Claude's Discretion.
  INSERT INTO scene_slides (
    scene_id,
    position,
    title,
    kind,
    design_json,
    duration_seconds
  ) VALUES (
    v_new_scene_id,
    0,                                                            -- single slide, position 0
    v_template.name,
    'default',                                                    -- align with template_library_slides default
    jsonb_build_object('svgContent', v_resolved_svg),             -- writes the exact shape SvgEditorPage:87-118 reads
    10                                                            -- align with template_library_slides.duration_seconds DEFAULT 10
  );

  -- NOTE: usage-count increment intentionally omitted.
  -- Mirrors migration 168:145-147 — deferred to Phase 175.

  RETURN v_new_scene_id;
END;
$$;
```

**Footer pattern** (168:153-157 — copy verbatim, substitute signature):
```sql
GRANT EXECUTE ON FUNCTION public.clone_svg_template_to_scene(uuid, text, text) TO authenticated;

COMMENT ON FUNCTION public.clone_svg_template_to_scene(uuid, text, text) IS
  'Phase 172.1: Atomic clone of an svg_templates row into scenes + scene_slides, writing the resolved SVG body (p_customized_svg > svg_templates.svg_content > RAISE) into scene_slides.design_json.svgContent. Mirrors clone_template_with_customization atomicity (single PL/pgSQL transaction). Access mirrors svg_templates SELECT RLS (migration 167) with super_admin bypass.';
```

**What explicitly diverges from 168 (checklist for planner):**
1. `v_template` type: `svg_templates%ROWTYPE` (not `template_library%ROWTYPE`)
2. Template fetch: uses `(tenant_id IS NULL OR created_by = v_user_id)` predicate (not `license IN (...)` license gate)
3. License gate: REMOVED entirely — no `template_enterprise_access` lookup (D-09)
4. SVG resolution block: NEW (no 168 analog) — three-way fallback per D-05
5. Scene INSERT `business_type`: `v_template.category` (not `v_template.industry`)
6. Scene INSERT `settings`: `jsonb_build_object(width, height, orientation)` (not `COALESCE(metadata, '{}')`)
7. Slide write: single INSERT (not `FOR v_slide IN SELECT ... FROM template_library_slides LOOP`)
8. Slide `design_json`: `jsonb_build_object('svgContent', v_resolved_svg)` (not conditional `jsonb_set`)
